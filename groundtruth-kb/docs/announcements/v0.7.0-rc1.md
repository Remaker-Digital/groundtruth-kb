# Announcing GroundTruth-KB v0.7.0-rc1

**Release date:** 2026-05-03 (release candidate 1)
**Theme:** Application/platform isolation
**Status:** rc1 — final tag and PyPI publication gated on Slice 8.5
CI-green VERIFIED

## What's new

GT-KB v0.7.0-rc1 delivers a strict separation between **GT-KB platform
artifacts** (the toolkit you install via `pip install groundtruth-kb`)
and the **adopter application code** that consumes it. Why this matters:

In v0.6.x, an adopter project mixed platform-managed files (`templates/`,
`.claude/rules/`, `.gtkb-state/`) with application-owned files (`src/`,
`tests/`, `scripts/`) in the same tree. Boundaries were documented but
not mechanically enforced. Drift was easy: an upgrade could overwrite
application code; an application import could reach into private platform
internals; a rebuild of the chroma overlay could lose user data sitting in
an unexpected location.

v0.7.0-rc1 makes the boundary mechanical via 12 doctor checks, registry
isolation enforcement, managed-only init defaults, isolation-aware upgrade
with rollback, and a 45-function clean-adopter test suite that exercises
the contract end-to-end.

## The isolation contract

For an adopter project rooted at `<project-root>/`:

- **Platform-managed tree:** files registered in
  `templates/managed-artifacts.toml` (managed by `gt project init` and
  `gt project upgrade`).
- **Adopter application tree:** under `applications/<app-name>/` (or
  the legacy mixed-tree mode opted into via `--include-app-tree`).
- **Harness state:** under `harness-state/<harness>/` per harness.
- **Bridge:** files under `bridge/` (workflow state); never embeds
  application source.
- **Database:** `groundtruth.db` carries platform records; application
  records flagged via `origin_project` field; never silently mixed.
- **Overlay:** `.groundtruth-chroma/` is a derived index; fully
  regeneratable from `groundtruth.db`. If lost, can be rebuilt; never
  authoritative.

Every boundary above has a `gt project doctor` check that enforces it.
Violations are ERROR-level; drift is WARN-level.

## Migration path for existing adopters

For an adopter on v0.6.1:

```bash
pip install --upgrade groundtruth-kb==0.7.0rc1
gt project upgrade --apply        # applies the new managed-only layout
gt project doctor                 # confirms isolation contracts pass
```

If `gt project doctor` reports isolation violations after upgrade, the
playbook is documented in
[`docs/architecture/isolation.md`](../architecture/isolation.md) and
exercised by 4 worked examples under
[`docs/examples/`](../examples/).

For adopters not yet ready to migrate, `gt project init --include-app-tree`
preserves the v0.6.x mixed-tree behavior. The 12 isolation doctor checks
can be scoped down in the project's doctor config until the migration is
complete.

## How to try it

Once Slice 8.5 records VERIFIED and the rc1 is tagged + published:

```bash
# Install the rc into a venv.
python -m venv .venv
source .venv/bin/activate    # Windows: .venv\Scripts\activate
pip install groundtruth-kb==0.7.0rc1

# Verify the installed CLI.
gt --version    # → 0.7.0rc1

# Discover the host_root the installed wheel binds to.
# (Per DELIB-S330-ISOLATION-017-SLICE8-INSTALL-UX-LIMITATION-ACK, the
# rc1 isolation contract requires --gt-kb-root to equal the install-derived
# _GT_KB_HOST_ROOT. For an installed wheel, this resolves to the venv root.)
HOST_ROOT=$(python -c "from groundtruth_kb.project.scaffold import _GT_KB_HOST_ROOT; print(_GT_KB_HOST_ROOT)")

# Scaffold an adopter project under <host_root>/applications/<name>.
mkdir -p "$HOST_ROOT/applications/MyAdopter"
gt project init MyAdopter \
    --profile local-only \
    --gt-kb-root "$HOST_ROOT" \
    --dir "$HOST_ROOT/applications/MyAdopter"

# Confirm the isolation contract.
gt project doctor --dir "$HOST_ROOT/applications/MyAdopter"
```

The clean-adopter test suite at `tests/clean_adopter/` is the executable
spec for what `gt project init` produces and what `gt project doctor`
asserts. Reading it gives a precise picture of the v0.7.0-rc1 contract.

### rc1 limitation: pip-install adopter UX

The command sequence above works but is awkward — adopter projects must
live under the install-derived `_GT_KB_HOST_ROOT` (the venv root for
pip-installed wheels), and the explicit `--gt-kb-root` + `--dir` arguments
are required. **v0.7.0 GA will simplify this.** The planned UX is
`gt project init MyAdopter --here` or `gt project init MyAdopter --target
<arbitrary-path>` so adopters can place projects anywhere with no
`--gt-kb-root` discovery step. Tracked as `GTKB-PIP-INSTALL-ADOPTER-UX-001` in the MemBase backlog
(`gt backlog list`); not blocking for the rc1 tag.

For now, if the explicit-host-root flow above feels heavy, the 4 worked
example trees under `groundtruth-kb/examples/` are pre-scaffolded adopters
you can copy and adapt without running `gt project init` at all.
