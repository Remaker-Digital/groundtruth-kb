# Deploy Agent Red

Deploy the application to staging or production following the build-deploy-verify pipeline.

**Arguments:** `$ARGUMENTS[0]` = environment (`staging` or `production`), `$ARGUMENTS[1]` = version (e.g., `v1.90.0`).

> **GOV-16 Deploy Gate:** All deployments require explicit owner approval. If the owner has not approved this deployment in the current session, STOP and ask for approval before proceeding.

## Phase A: Pre-Flight

Before building, verify ALL of the following. If ANY check fails, STOP and report.

1. **A.1 Version Bump** -- Verify `PRODUCT_VERSION` in `src/multi_tenant/api_versioning.py` matches `$1` (without `v` prefix).

2. **A.2 Protected Behavior Regression Gate** -- Run the 11 PB grep assertions. See `references/protected-behaviors.md` for the full assertion table and commands. **If ANY fails, STOP.**

3. **A.3 No Uncommitted Changes** -- `git diff --stat src/ admin/ widget/` must be empty.

4. **A.4 Unit Tests Pass** -- `.\scripts\run-tests-thermal-safe.ps1 -Workers 4 -CoolDown 30 -SkipLive` must report OVERALL: PASS.

5. **A.5 Security Scan** -- Run `/check-security src/` to scan for OWASP Top 10 vulnerabilities across the source tree. Review any P0/P1 findings before proceeding.

6. **A.6 Code Quality Review** -- Run `/quick-review` on changes since last deploy tag to verify no regressions, logic errors, or governance violations in the diff.

## Phase B: Build

See `references/environment-config.md` for full build commands (B.0-B.6).

1. Clear `VITE_API_URL` in all 3 admin `.env.local` files
2. Build 3 admin SPAs (`standalone`, `shopify`, `provider`) + widget
3. Verify artifact freshness (all 4 dist outputs < 5 min old)
4. ACR build: `az acr build --registry acragentredeastus --image api-gateway:$1 --no-logs --file Dockerfile .`

## Phase C: Deploy

```bash
az containerapp update --name <CONTAINER_APP> --resource-group Agent-Red --image acragentredeastus.azurecr.io/api-gateway:$1
```

Wait 90-180s (staging needs longer for scale-from-zero), then verify `/health` returns 200 with correct `product_version`. See `references/environment-config.md` for environment-specific parameters.

## Phase D: Post-Deploy Verification

See `references/environment-config.md` for full verification commands (D.1-D.4).

1. Version header matches deployed version
2. All 3 admin SPAs return HTTP 200
3. Widget accessible (> 1,000 bytes)
4. Upgrade verification (35 assertions per tenant)

## Post-Deploy Checklist

1. Update `memory/MEMORY.md` with new image tag, ACR run ID, and revision name
2. Record deployment in Knowledge Database
3. If production: update wiki deployment history

## Rollback

If post-deploy verification fails, revert to previous image and run Tier 0 regression tests. See `references/environment-config.md` for rollback command.
