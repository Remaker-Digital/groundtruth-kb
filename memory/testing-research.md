# Testing AI-Generated Code — Research & Approaches

> Compiled 2026-03-21 from industry sources. This is operational memory for Claude, not canonical project knowledge.

## The Problem Statement

Agent Red faces four systemic defect classes that testing alone cannot solve:

1. **Architectural amnesia** — Claude forgets past design decisions and undoes them (e.g., disabling SLIM/MCP)
2. **Dead UI** — Frontend elements rendered but not connected to backend data sources
3. **Layout drift** — UI changes that violate prior design specs recorded in KB
4. **Test weakening** — Tests removed or assertions softened to achieve pass rates

All four share a root cause: **the AI agent's context window is finite, session memory is lossy, and the path of least resistance to a passing build is often destructive.**

---

## Industry Findings

### AI Code Has Higher Defect Rates
- AI code has **1.7× more bugs** than human-written code (Qodo State of AI Code Quality 2025)
- **1.64× higher** maintainability errors in AI-generated codebases
- GitClear: refactoring collapsed from 25% to under 10% in AI-heavy codebases
- CodeRabbit: AI code had **2.74× more security vulnerabilities**
- METR: developers took **19% longer** overall despite initial coding speed gains

### AI Code Fails Differently
Three failure patterns unique to AI-generated code (Testkube):

1. **Logic Drift** — Code is syntactically correct but violates invisible business rules (e.g., discount stacking constraints buried in comments, not formal specs)
2. **Dependency Mismatches** — AI suggests packages from training snapshots, not current vulnerability databases
3. **Regression Roulette** — Optimized code passes immediate tests but breaks rarely-executed paths (e.g., NULL handling changes that break monthly reports)

### The "Dead UI" Problem
AI treats backend as a raw data source, creating the simplest possible implementation. Results in:
- Forms that render but don't submit
- Modals that open but have no save handler
- Components that fetch all data and filter client-side instead of using paginated APIs
- Code that compiles and looks functional but has broken frontend-backend interactions

## Proven Approaches

### 1. Pre-Commit Hook Architecture (Three Layers)
Source: [I Built the Guardrails Into the Repo, Not the Prompt](https://dev.to/wilddog64/i-built-the-guardrails-into-the-repo-not-the-prompt-4n3l)

**Key insight: "Prompts encode intent. Hooks encode enforcement."**

| Layer | Type | Purpose |
|-------|------|---------|
| Layer 1 | Deterministic | Structural rules (e.g., subtree guards, import restrictions) |
| Layer 2 | Agent Audit | Detect removed tests, unsafe patterns, excessive complexity |
| Layer 3 | AI-Powered | Review changes against architectural rules in markdown |

**Critical detail:** Pre-commit enforcement happens *before the commit exists*. The agent's work is stopped at the point where it can still fix the issue. The `--no-verify` escape hatch is measurable — its use is itself a signal.

### 2. Coverage Ratchet
Source: [jest-ratchet](https://github.com/markis/jest-ratchet), SonarQube ratcheting

Coverage thresholds only go up, never down. Every time coverage improves, the threshold automatically increases. Prevents the "remove tests to pass" anti-pattern by making reduction a build failure.

**Implementation:** Store coverage baseline in a committed file. Pre-commit hook compares current coverage to baseline. If current < baseline, reject. If current > baseline, update baseline.

### 3. Test Assertion Ratchet (our invention — no standard tool exists)
Extend the coverage ratchet to assertion counts per test file. A committed baseline JSON maps `test_file → assertion_count`. If a commit reduces assertions in any file, the hook rejects it. Tests can be added but never weakened.

### 4. Architectural Decision Records (ADR) as Executable Rules
Source: [Archgate CLI](https://github.com/archgate/cli)

Each ADR can have a companion `.rules.ts` file that exports automated checks. `archgate check` scans the codebase against these rules and reports violations. Integrates with CI and pre-commit hooks.

**Key concept: "fitness functions"** — objective automated checks that verify architectural decisions are maintained. A decision record documents the decision; a fitness function assures it.

### 5. Meta's JiT Testing (Just-in-Time Tests)
Source: [Meta Engineering Blog, Feb 2026](https://engineering.fb.com/2026/02/11/developer-tools/the-death-of-traditional-testing-agentic-development-jit-testing-revival/)

Tests generated by LLMs on-the-fly for each code change. A "Catching JiTTest" specifically finds regressions introduced by a diff:

1. LLM analyzes the diff
2. Infers developer intent
3. Generates mutations that simulate possible faults
4. Creates tests that would catch those mutations
5. Filters false positives
6. Reports only genuine regressions

**Key advantage:** Tests don't reside in the codebase. No maintenance cost. Only triggers human review when a bug is actually caught. Diff-aware generation produces **4× more catches** than hardening workflows.

### 6. Contract Testing (Frontend ↔ Backend)
Source: [Pact](https://docs.pact.io/)

Consumer-driven contract testing: the frontend declares what it expects from the API. The API verifies it can satisfy those expectations. Prevents the "dead UI" problem by making the contract explicit and machine-verifiable.

### 7. Snapshot Testing for Data Transformations
Source: [Testkube](https://testkube.io/blog/testing-ai-generated-code)

Serialize critical data transformations to reference snapshots. Any output change gets flagged for review. Catches "regression roulette" where refactored code silently changes edge case behavior.

### 8. Property-Based Testing (Hypothesis)
Source: [Hypothesis](https://github.com/HypothesisWorks/hypothesis), [Agentic PBT paper](https://arxiv.org/html/2510.09907v1)

Define invariants that must hold for ALL inputs. Hypothesis generates hundreds of random inputs and edge cases automatically. When a failure is found, it remembers the input and retests in future runs — **built-in regression prevention**.

Agent Red already has 46 property tests (SPEC-1843). Expanding these to cover tier gating, rate limiting, and entitlement logic would catch the "logic drift" class of AI defects.

### 9. Mutation Testing (mutmut)
Source: [Meta ACH](https://engineering.fb.com/2025/09/30/security/llms-are-the-key-to-mutation-testing-and-better-compliance/)

Deliberately introduce faults. If tests don't catch them, the tests are too weak. Meta's ACH uses LLMs to generate *realistic* mutants (not random syntax changes), producing targeted tests that catch those mutants.

Agent Red already has mutmut configured (SPEC-1842). Running it regularly and tracking the mutation score with a ratchet would prevent test weakening.

## Approaches Specific to Agent Red's Problems

### Problem 1: Architectural Amnesia (undoing SLIM, MCP, etc.)

| Approach | Mechanism | Effort |
|----------|-----------|--------|
| **Pre-commit grep guards** | Hook greps for mandatory patterns (SLIM imports, MCP registry) — rejects if missing | Low |
| **Protected behavior assertions** | PB-* specs already exist. Run them as pre-commit checks, not just session-start | Low |
| **ADR fitness functions** | Each architectural decision gets a machine-verifiable check | Medium |
| **Diff-aware intent checking** | Before commit, check if the diff REMOVES any pattern that a spec says MUST exist | Medium |

### Problem 2: Dead UI (disconnected frontend-backend)

| Approach | Mechanism | Effort |
|----------|-----------|--------|
| **Contract testing** | SPA declares expected API shape. Backend verifies it can satisfy. Catches disconnected forms. | High |
| **Form handler audit** | Pre-commit hook checks that every `<form>` or `onSubmit` in TSX has a matching `apiFetch` call | Medium |
| **E2E Playwright tests** | Actually click buttons, fill forms, verify server response. The e2e_live suite does this. | Already exists |
| **Static analysis** | Custom ESLint rule: modal with inputs must have a submit handler that calls an API | Medium |

### Problem 3: Layout Drift (unauthorized UI changes)

| Approach | Mechanism | Effort |
|----------|-----------|--------|
| **TSX structural snapshot** | Hash the component tree structure (not rendered pixels). Any structural change = review required. | Medium |
| **Pre-commit TSX change gate** | Any change to a `.tsx` file requires a SPEC-ID in the commit message | Low |
| **Visual regression testing** | Screenshot comparison (Playwright + pixelmatch). Catches unintended visual changes. | High |
| **Component manifest** | Committed JSON listing all pages, their components, and layout structure. Diff against it. | Medium |

### Problem 4: Test Weakening (removing/softening tests)

| Approach | Mechanism | Effort |
|----------|-----------|--------|
| **Assertion count ratchet** | Committed baseline of assertion counts per file. Decrease = rejection. | Low |
| **Test file deletion guard** | Pre-commit hook rejects deletion of any `test_*.py` file | Low |
| **Coverage ratchet** | Coverage baseline that only increases | Low |
| **Mutation score ratchet** | mutmut score baseline that only increases | Medium |

---

## Recommended Implementation Priority

**Phase 1 — Immediate (pre-commit hooks, no infrastructure changes):**
1. Assertion count ratchet (prevents test weakening)
2. Test file deletion guard (prevents test removal)
3. Architectural grep guards (prevents SLIM/MCP removal)
4. TSX commit gate (requires SPEC-ID for UI changes)

**Phase 2 — Short-term (test infrastructure):**
5. Coverage ratchet (prevents coverage regression)
6. Protected behavior assertions in pre-commit (not just session-start)
7. Expand property-based tests to all critical business logic

**Phase 3 — Medium-term (contract and structural testing):**
8. Frontend-backend contract testing
9. Component structural snapshots
10. Mutation score ratchet

**Phase 4 — Long-term (JiT and AI-aware):**
11. Diff-aware regression test generation (Meta JiT pattern)
12. AI-powered architectural review (Layer 3 hooks)
