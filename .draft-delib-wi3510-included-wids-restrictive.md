# Owner Decision — PAUTH `included_work_item_ids` Restrictive Semantics

## Context

`DELIB-2547` (S379, "Reduce friction, keep gates") deferred the canonical
directionality decision for the `included_work_item_ids` field on a project
authorization (PAUTH). Two governance gates evaluate PAUTH scope divergently:

- Implementation-start gate (`scripts/implementation_authorization.py`): a work
  item is authorized if it is in `included_work_item_ids` OR is an active member
  of the cited project (additive / loose).
- Bridge Write-time gate (`.claude/hooks/bridge-compliance-gate.py`): a work item
  is authorized only if it is an active project member AND (the included list is
  empty OR the work item is listed) (restrictive / strict).

This divergence is the friction `WI-3510` (PROJECT-GTKB-RELIABILITY-FIXES) set out
to reconcile. Reconciliation required the owner to choose the canonical semantics.

## Options Presented (AskUserQuestion)

1. Restrictive — the non-empty `included_work_item_ids` list is the authoritative
   scope at both gates (empty list falls back to active project membership).
2. Additive — a work item is in scope if it is a project member OR listed, at both
   gates.
3. Defense-in-depth — keep today's divergence by design and document the rationale.

## Owner Decision

**Restrictive — the list is the authoritative scope.**

- A non-empty `included_work_item_ids` list defines the authorized work-item scope
  at BOTH the bridge Write-time gate and the implementation-start gate.
- A work item is in scope if and only if it appears in the list.
- An empty `included_work_item_ids` list falls back to active project membership.
- `excluded_work_item_ids` continues to exclude regardless of list or membership.

## Rationale

- Matches established usage: most active PAUTHs already use a non-empty
  `included_work_item_ids` list as the intended scope.
- Honors `GOV-PROJECT-IMPLEMENTATION-AUTHORIZATION-001`: active backlog/project
  membership alone is not implementation authorization; the explicit list is the
  authorization surface.
- Eliminates the gate divergence by tightening the looser implementation-start gate
  to the restrictive reading, rather than relaxing the Write-time gate.

## Consequence / Next Steps

- Draft and present for approval a new design constraint
  (`DCL-PAUTH-INCLUDED-WORK-ITEM-IDS-RESTRICTIVE-001`) encoding this semantics with
  machine-checkable assertions.
- After DCL approval and insertion, file a substantive implementation REVISED on
  `bridge/gtkb-reconcile-included-work-item-ids-semantics` citing the DCL and
  deriving tests from it, reconciling `scripts/implementation_authorization.py` to
  the restrictive reading and confirming `.claude/hooks/bridge-compliance-gate.py`
  matches.
