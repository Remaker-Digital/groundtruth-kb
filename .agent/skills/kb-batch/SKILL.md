# KB Batch Operations

Perform bulk Knowledge Database operations with strong safety controls.

**Arguments:**
- `$ARGUMENTS[0]` = operation: `resolve-wis`, `promote-specs`, `retire-specs`
- `$ARGUMENTS[1]` = comma-separated IDs (e.g., `WI-1663,WI-1664,WI-1665`)
- `--execute` flag required to actually apply changes (default: dry-run)

## Safety Controls

1. **Mandatory dry-run**: Without `--execute`, only shows what would change
2. **GOV-15 gate**: Defect/regression WIs require `owner_approved=True` — script prompts for confirmation
3. **50-item maximum**: Rejects batches larger than 50 to prevent runaway operations
4. **Validation**: All IDs verified to exist before any changes
5. **Change reason**: Auto-generated with batch operation context

## Operations

### resolve-wis
Resolve work items: sets `resolution_status=resolved`, `stage=resolved`.
For defect/regression origin WIs, requires owner approval confirmation.

### promote-specs
Promote spec status: `specified` → `implemented` or `implemented` → `verified`.
Architecture-layer specs (ADR/DCL) require assertions per Option B gate.

### retire-specs
Set spec status to `retired` with supersession reason.

## Example

```
/kb-batch resolve-wis WI-1663,WI-1664,WI-1665           # dry-run
/kb-batch resolve-wis WI-1663,WI-1664,WI-1665 --execute  # apply
/kb-batch promote-specs SPEC-1706,SPEC-1707 --execute
/kb-batch retire-specs SPEC-1789 --execute
```
