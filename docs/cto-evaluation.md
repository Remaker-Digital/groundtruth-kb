# CTO Evaluation Guide

This guide is the shortest path for a technical evaluator who wants to prove
that GroundTruth-KB can be installed from pip, create a usable project, expose
the local dashboard, and explain the operating model without private project
context.

## Evaluation Goal

The evaluation succeeds when a clean workstation can:

1. Install `groundtruth-kb` with pip.
2. Create and verify a project with `gt project init` and `gt project doctor`.
3. Generate the operations dashboard with `gt dashboard init`.
4. Install and start local Grafana with `gt dashboard install` and
   `gt dashboard start`.
5. Run the session start/wrap-up cycle with clear Prime Builder and Loyal
   Opposition role boundaries.

## Install From Pip

Use Python 3.11 or newer.

```powershell
python --version
python -m pip install --upgrade pip
python -m pip install groundtruth-kb
gt --version
```

Create a local evaluation project:

```powershell
gt project init cto-gtkb-eval --profile local-only --no-seed-example --no-include-ci
cd cto-gtkb-eval
gt project doctor
gt assert
```

The `dual-agent` profile adds Prime Builder / Loyal Opposition bridge files and
agent instructions. Start with `local-only` if the first question is package
installation; rerun with `dual-agent` when evaluating the collaboration model.

## Operations Dashboard

The dashboard is generated by the pip package. It does not depend on
repository-local scripts.

```powershell
gt dashboard init
```

This creates:

- `.groundtruth/dashboard/gtkb-dashboard.sqlite` - generated reporting DB
- `.groundtruth/dashboard/grafana/provisioning/` - Grafana datasource and
  dashboard provider configuration
- `.groundtruth/dashboard/grafana/dashboards/groundtruth-kb-dashboard.json` -
  dashboard definition

Install and start the local dashboard:

```powershell
gt dashboard install
gt dashboard start
```

Open:

```text
http://127.0.0.1:3000/d/groundtruth-kb/groundtruth-kb-dashboard
```

Refresh manually when needed:

```powershell
gt dashboard refresh
```

Stop local dashboard processes:

```powershell
gt dashboard stop
```

The dashboard reports current metrics, setup steps, integration status,
third-party service inventory, active actions, risks, and data freshness.

## Session Start And Wrap-Up Cycle

A GroundTruth session has three phases:

| Phase | Required behavior | Evidence |
|---|---|---|
| Start | Load role instructions, project state, MemBase status, bridge queue, and top priorities. | Startup disclosure, file bridge scan count, dashboard health. |
| Execute | Make scoped changes, record specs/work items/deliberations where appropriate, and run assertions during meaningful checkpoints. | Git diff, MemBase rows, command output, bridge review files. |
| Wrap up | Refresh dashboard data, run final checks, record what changed and what is next, then commit with a session-aware message. | `gt dashboard refresh`, `gt assert`, test logs, commit hash. |

The package provides the durable artifacts and commands. Individual adopter
repos may add harness-specific hooks for automatic startup disclosure or wrap-up
prompts, but those hooks are project configuration rather than a pip dependency.

## Prime Builder And Loyal Opposition

GroundTruth separates implementation from evaluation:

| Role | Purpose | Normal output |
|---|---|---|
| Prime Builder | Builds the system, updates specs, runs tests, performs wrap-up. | Code, specs, work items, bridge proposals, commits. |
| Loyal Opposition | Reviews plans/code/configuration and challenges assumptions with evidence. | GO, NO-GO, or VERIFIED review reports. |

The owner assigns the active Prime Builder. The role is not tied to one vendor
or model. A common scaffold maps Claude Code to Prime Builder and Codex to
Loyal Opposition, but the important contract is role behavior: one harness owns
implementation, the independent harness owns evidence-based review.

When an owner temporarily designates a different active builder, Prime Builder
permissions and restrictions follow the role assignment. Loyal Opposition
restrictions apply only to the reviewer role, not to the active builder.

## Third-Party Service Inventory

GroundTruth-KB deliberately keeps third-party services explicit:

| Service | Required for | Credential handling |
|---|---|---|
| PyPI | Installing `groundtruth-kb` with pip. | Public package install needs no project credential. |
| GitHub | Source hosting, PRs, generated GitHub Actions, CodeQL, Dependabot. | GitHub tokens/accounts are managed by the adopter. |
| SonarCloud | Hosted quality gate for this package repo. | Optional for adopters; use an internal equivalent if required. |
| Grafana OSS | Local operations dashboard. | Local OSS runtime needs no cloud credential. |
| frser SQLite datasource | Grafana reads `gtkb-dashboard.sqlite`. | Installed by `gt dashboard install`; no credential. |
| ChromaDB | Optional semantic search for deliberations. | Local embedded index; install with `groundtruth-kb[search]`. |
| Claude Code | Common Prime Builder harness. | Auth is managed outside GroundTruth-KB. |
| OpenAI Codex | Common Loyal Opposition harness. | Auth is managed outside GroundTruth-KB. |
| MkDocs Material | Documentation site rendering. | Optional docs dependency. |

Credential lifecycle is outside GroundTruth-KB scope. The toolkit scaffolds
instructions, stores local governance data, and reports integration status; it
does not rotate or manage external service secrets.

## Maintainer Release Checks

Before pushing a package update intended for CTO evaluation, maintainers should
run:

```powershell
python -m pytest -q --tb=short
python -m ruff check .
python -m ruff format --check .
python -m pip install build
python -m build
python -m pip install --force-reinstall .\dist\groundtruth_kb-*.whl
gt --version
```

Then create a fresh project outside the repository and run:

```powershell
gt project init cto-smoke --profile local-only --no-seed-example --no-include-ci
cd cto-smoke
gt project doctor
gt dashboard init
gt dashboard refresh
```

Run `gt dashboard install/start` on the evaluator workstation when a live
Grafana check is required. In locked-down corporate environments, install
Grafana and the SQLite plugin through approved enterprise channels and pass
`--grafana-home` to the dashboard commands.
