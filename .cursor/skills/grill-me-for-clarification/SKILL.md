# /grill-me-for-clarification

Conduct a structured, dependency-ordered clarification interview of the owner
about an explicitly-scoped plan or design, until you reach shared
understanding. Every resolved branch is persisted as a durable artifact, so the
interview survives session boundaries.

This skill operationalizes the Prime Builder interrogative default
(`DELIB-S324-PB-INTERROGATION-DIRECTIVE`) and the operating-model goal of
reducing the owner's role to specifications, clarifications, and decisions.

## When to invoke

Use this skill when the owner wants to:

- stress-test a plan or design before committing to it,
- be grilled on their requirements, specifications, or directives,
- reach shared understanding on a multi-decision area before implementation,
- or explicitly says "grill me for clarification".

## Governing artifacts

- Requirement: `SPEC-INTAKE-1262c1` / `INTAKE-45c006c4` (the operative
  owner-confirmed requirement text).
- Persistence backends: the `gtkb-decision-capture` and `gtkb-spec-intake`
  skills.
- Decision channel: `AskUserQuestion` only — never a prose decision-ask (per
  the AUQ-only enforcement stack).

## Non-goals

This skill does NOT:

- write or modify code,
- file bridge proposals,
- promote specs beyond `gtkb-spec-intake` confirm,
- create work items beyond `gtkb-spec-intake` confirm's deterministic
  behavior.

Its output is durable decisions and a shared-understanding summary — not
implementation.

## Phase 0 — Scope intake (required)

The skill requires an explicit scope argument naming the plan, design, project
set, spec, or backlog range to interview. There is no default scope.

If the skill is invoked with no argument, STOP and ask the owner for the
scope. Do not proceed, and do not substitute a guessed scope.

## Phase 1 — Build the decision tree

Before asking the owner anything, explore the in-scope artifacts read-only:
MemBase specs, work items, and projects; bridge threads; the Deliberation
Archive; rule files; and code.

Enumerate every open decision point in scope. For each, record its
dependencies — which other decisions must be resolved before it.

For every question the codebase can answer, answer it by exploration and
exclude it from the owner-question set. The owner is asked only what genuinely
needs an owner decision.

Emit the decision tree in dependency (topological) order: a decision is only
asked after the decisions it depends on are resolved.

## Phase 2 — Traverse one question at a time

Walk the tree in dependency order. For each unresolved decision:

1. Re-check the codebase first — a prior answer or fresh exploration may have
   made the question answerable without the owner.
2. Ask exactly ONE question per turn via `AskUserQuestion`.
3. Present the top three recommended answers, recommended option first
   (label it "(Recommended)").
4. When a new decision point is discovered mid-traversal, insert it into the
   tree at its correct dependency position rather than asking it out of order.

Never batch multiple decisions into one turn. Never ask a decision before its
dependencies are resolved.

## Phase 3 — Persist each resolution (checkpoint)

Immediately after each owner answer, classify and route the resolution:

- A pure decision (a policy, scope, or design selection) → `gtkb-decision-capture`
  (a Deliberation Archive record, `outcome=owner_decision`).
- A requirement-shaped resolution (it states what the system must do) →
  `gtkb-spec-intake` (capture, then owner-confirmed promotion to a spec).

Checkpointing per branch is what makes a long interview resumable: if the
session ends mid-tree, every answered branch is already a durable artifact and
the next session resumes from the unresolved branches.

## Phase 4 — Shared-understanding summary

When the tree is exhausted (or at a session boundary), produce a consolidated
summary:

- each resolved branch → its resolution → the `DELIB-` / `SPEC-` ID that
  persisted it,
- each deferred branch → the explicit trigger condition for revisiting it.

The summary is the shared-understanding artifact: it lets the owner and any
future session see the whole resolved tree at a glance.
