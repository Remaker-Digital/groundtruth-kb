# DCL-PAUTH-INCLUDED-WORK-ITEM-IDS-RESTRICTIVE-001 — `included_work_item_ids` is restrictive (authoritative) scope

## Constraint

A project authorization's (PAUTH) `included_work_item_ids` field defines the
authoritative work-item scope of that authorization. Both governance gates — the
bridge Write-time gate (`.claude/hooks/bridge-compliance-gate.py`,
`_wi_project_membership_gap`) and the implementation-start gate
(`scripts/implementation_authorization.py`, `validate_project_authorization_row`) —
MUST apply identical semantics for a cited (authorization, project, work item):

1. If `excluded_work_item_ids` contains the work item: OUT OF SCOPE (blocked),
   regardless of any other field.
2. Else if `included_work_item_ids` is NON-EMPTY: IN SCOPE if and only if the work
   item appears in `included_work_item_ids`. The list is authoritative; active
   project membership is neither required nor sufficient on its own.
3. Else (`included_work_item_ids` is EMPTY): IN SCOPE if and only if the work item
   is an active member of the cited project.

## Canonical truth table (non-empty `included_work_item_ids`)

| active project member | in included list | authorized |
|---|---|---|
| yes | yes | YES |
| yes | no  | NO — membership alone is not authorization |
| no  | yes | YES — the explicit list is authoritative |
| no  | no  | NO |

## Rationale

- Owner decision `DELIB-20266083` (2026-06-25 AskUserQuestion): restrictive — the
  list is the authoritative scope. Resolves the `DELIB-2547` deferral.
- Matches live usage: 229 of 235 active PAUTHs carry a non-empty
  `included_work_item_ids` list, and 77 listed (work-item, PAUTH) pairs are not
  active project members. The list-authoritative reading keeps those 77 authorized;
  additionally requiring active membership would block them — a regression.
- Honors `GOV-PROJECT-IMPLEMENTATION-AUTHORIZATION-001`: active project/backlog
  membership alone is not implementation authorization.
- Eliminates the prior gate divergence (impl-start additive `listed OR member` vs.
  Write-time `member AND listed`) by making both gates apply this reading.

## Enforcement / Assertions

- A1: implementation-start gate applies the semantics above (tightens current
  `listed OR member`: a member that is not listed is no longer authorized when the
  included list is non-empty).
- A2: bridge Write-time gate applies the identical semantics (a non-empty list is
  authoritative, so a listed work item is authorized even without a recorded active
  membership; an empty list falls back to active membership).
- A3: `excluded_work_item_ids` exclusion takes precedence at both gates.
- A4: a spec-derived test asserts both gates agree across the truth table.
