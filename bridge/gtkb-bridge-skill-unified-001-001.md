## Claim

Expose the full bridge protocol surface as a single canonical "bridge" skill ("propose, scan, respond, verify, lifecycle states, status") that appears and behaves identically across Claude Code and Codex harnesses. This eliminates per-platform divergence in HOW agents are taught to operate the bridge, while delegating execution to harness-agnostic scripts/CLI.

## Risk / Rollback

Risk surface:

- **Skill name collision**: `bridge` is a short, generic name. Verify no existing skill or built-in directive uses it. Mitigation: T-1-canonical-present + harness-parity check.
- **Existing per-action skill obsolescence**: if `bridge` subsumes `bridge-propose`, existing references to the latter may break. Mitigation: Slice 1 step A2 explicitly addresses coexistence vs supersession; if superseded, the registry's `parity_class` field can mark legacy skills as deprecated without removing them.
- **CLI dependency uncertainty**: Slice 3 deferral means the unified skill initially delegates to existing scripts (`scripts/bridge_propose.py`, etc.) rather than a clean `gt bridge` CLI. The skill body needs to document the CLI invocations honestly — even if the underlying delegation is currently to per-script entries. Mitigation: skill body documents the CONCEPTUAL surface; underlying scripts can be refactored over time without changing the skill content.

Rollback: revert the canonical SKILL.md + registry entry + regenerate adapters. Existing per-action skills (`bridge-propose`, etc.) continue functioning as today.
