# NO-GO: GroundTruth-KB Deliberation CLI Proposal Review

**Reviewer:** Codex Loyal Opposition
**Date:** 2026-04-14

**Target repo inspected:** `E:\Claude-Playground\CLAUDE-PROJECTS\groundtruth-kb`

## Verdict

NO-GO.

The proposal targets the right gap: the Deliberation Archive has a complete
Python API but only one maintenance CLI command. However, the proposed CLI
contract conflicts with the existing API in several concrete places. The most
important issues are the `--upsert`/`--id` mismatch and the proposal's plan to
make `search` fail without ChromaDB even though the stable Python API already
supports SQLite text fallback.

Prime should revise the proposal before implementation.

- Current docs document only the rebuild-index deliberation command:

## Findings

### 1. HIGH - `add --upsert --id` conflicts with `upsert_deliberation_source`

**Claim in proposal:** `gt deliberations add [--upsert]` always requires
`--id <DELIB-ID>`, and `--upsert` switches to
`upsert_deliberation_source`.

**Evidence:**

- `insert_deliberation` requires an explicit `id`:
  `src/groundtruth_kb/db.py:3917-3926`.
- `upsert_deliberation_source` has no explicit `id` parameter:
  `src/groundtruth_kb/db.py:4012-4018`.
- On a new source, `upsert_deliberation_source` generates `new_id` itself and
  calls `insert_deliberation(id=new_id, ...)`:
  `src/groundtruth_kb/db.py:4035-4051`.
- Existing tests assert this generated-ID behavior:
  `tests/test_deliberations.py:519-530` expects the first upsert to create
  `DELIB-0001`; `tests/test_deliberations.py:576-596` expects auto-increment.

**Risk/impact:** A literal implementation that passes the CLI `--id` into
`upsert_deliberation_source` will either raise a duplicate `id` argument error
or silently ignore the user's requested ID. Either behavior is bad for a
developer-facing CLI.

**Required action:** Revise the CLI contract. Acceptable options:

- Make `--id` required only for append-only `add`, and mutually exclusive with
  `--upsert`.
- Split idempotent writes into `gt deliberations upsert`, where IDs are
  generated by the DB and the command reports the chosen ID.
- Change the Python API first to support caller-supplied IDs for idempotent
  upsert, with tests proving the semantics.

The revised proposal must state the exact behavior and update tests
accordingly.

### 2. HIGH - Proposed `search` failure without ChromaDB contradicts the stable API

**Claim in proposal:** `gt deliberations search` requires the `[search]` extra;
without ChromaDB it exits 1 with install guidance. Test T13 expects this
failure.

**Evidence:**

- `search_deliberations` is documented in code as semantic search with SQLite
  LIKE fallback when ChromaDB is unavailable or irrelevant:
  `src/groundtruth_kb/db.py:4301-4306`.
- The fallback implementation queries `current_deliberations` with LIKE and
  returns `search_method="text_match"`:
  `src/groundtruth_kb/db.py:4357-4374`.
- The method guide describes the same fallback:
  `docs/method/13-deliberation-archive.md:104-128`.
- Tests cover base fallback behavior:
  `tests/test_deliberations.py:654-717` and
  `tests/test_deliberations.py:752-779`.

**Risk/impact:** Requiring ChromaDB for the CLI search path would make the
base install less capable than the Python API and would hurt the same
CLI-first adoption path the proposal is trying to improve.

**Required action:** Revise `gt deliberations search` to call
`KnowledgeDB.search_deliberations()` and return text-match fallback results in
base installs. If the CLI needs a semantic-only mode, add an explicit flag such
as `--semantic-only` and test that flag separately. Update T13 to expect
fallback results or no matches, not an install error.

### 3. MEDIUM - Link error tests require validation that the DB methods do not perform

**Claim in proposal:** `test_link_nonexistent_spec_errors` should fail when a
spec does not exist, while the command remains a thin wrapper.

**Evidence:**

- `deliberation_specs` and `deliberation_work_items` tables have no foreign key
  constraints: `src/groundtruth_kb/db.py:352-364`.
- `link_deliberation_spec` performs `INSERT OR REPLACE` directly:
  `src/groundtruth_kb/db.py:4132-4140`.
- `link_deliberation_work_item` performs `INSERT OR REPLACE` directly:
  `src/groundtruth_kb/db.py:4142-4150`.

**Risk/impact:** A thin wrapper will happily link nonexistent specs, work
items, or deliberations. The proposed error-path tests will not pass unless
the CLI adds validation logic or the DB layer changes.

**Required action:** Choose and document one contract:

- CLI validates `get_deliberation`, `get_spec`, and `get_work_item` before
  linking and exits 1 on missing targets.
- DB methods are changed to validate or enforce foreign keys, with migration
  and tests.
- The proposal drops nonexistent-target error tests and explicitly allows
  forward links.

### 4. MEDIUM - Required DB fields are optional in the CLI surface without defined defaults

**Claim in proposal:** `--summary`, `--changed-by`, and `--change-reason` are
optional CLI flags.

**Evidence:** `insert_deliberation` requires `summary`, `changed_by`, and
`change_reason` as positional arguments:
`src/groundtruth_kb/db.py:3917-3926`.

**Risk/impact:** The minimal add test cannot pass unless the CLI invents
defaults. Silent defaults can also weaken audit quality if every row says only
`changed_by=cli`.

**Required action:** The revised proposal must either make these flags required
or define explicit defaults and rationale. At minimum, require
`--change-reason` or default it to a deterministic string such as
`gt deliberations add`; define `--changed-by` default from an existing local
config/user convention if one exists.

### 5. MEDIUM - The test count and scope need to be reconciled after the contract changes

The proposal says "~14 tests" in one place, lists T1-T18 elsewhere, and later
expects pytest growth from 600 to 618. That is fixable, but the final test
matrix should align with the revised contracts above.

**Required action:** In the revision, provide a stable test list that covers:
append-only add, idempotent upsert semantics, base search fallback, optional
semantic search, link validation contract, JSON output, and docs coverage.
