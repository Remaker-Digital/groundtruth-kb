## Findings

### 1. HIGH - The proposed coverage command uses an invalid report type

**Claim in proposal:** Run
`pytest --cov=groundtruth_kb --cov-branch --cov-report=html:... --cov-report=md:docs/reports/v0.4-baseline/coverage.md`.

**Evidence:**

- `python -m pytest --help` lists coverage report types as `term`,
  `term-missing`, `annotate`, `html`, `xml`, `json`, `markdown`,
  `markdown-append`, and `lcov`. It does not list `md`.
- Running
  `python -m pytest tests/test_cli.py::TestCLI::test_version --cov=groundtruth_kb --cov-branch --cov-report=md:$env:TEMP\gtkb-cov-md-test.md -q`
  failed during option parsing with:
  `invalid choice: "md:..."`.

**Risk/impact:** The first report generation step fails before any baseline is
produced.

**Required action:** Revise the command to use the supported report type:

```bash
python -m pytest --cov=groundtruth_kb --cov-branch \
  --cov-report=markdown:docs/reports/v0.4-baseline/coverage.md \
  --cov-report=html:docs/reports/v0.4-baseline/coverage-html
```

If HTML is not committed, generate it outside the repo or add a clear cleanup
step.

### 2. HIGH - The proposed interrogate JSON command is invalid

**Claim in proposal:** Run
`interrogate src/groundtruth_kb/ --verbose 2 --fail-under 0 --output-format json`
and parse JSON.

**Evidence:**

- `python -m interrogate --help` from a temp install of `interrogate>=1.7`
  lists `--output FILE`, but no `--output-format` option.
- Running the proposed command with that temp install failed with:
  `Error: No such option: --output-format (Possible options: --badge-format, --output)`.
- The current local Python environment did not have interrogate installed
  until I installed it in a temp target for validation:
  `python -m interrogate --help` initially returned `No module named interrogate`.

**Risk/impact:** `docstrings.md` cannot be generated by the proposed flow, and
any "small analysis script" based on JSON input has no valid input.

**Required action:** Revise the docstring audit to use a supported source:

- parse `interrogate -vv --fail-under 0 --output <file>` text output, or
- use the `interrogate` Python API explicitly, with a short script committed
  or included in the post-implementation report, or
- choose a different tool that actually emits JSON.

The report must record the exact interrogate version used.

### 3. MEDIUM - The mypy plan is underspecified for an ad-hoc measurement

**Claim in proposal:** Run `mypy --strict src/groundtruth_kb/` "in non-strict
mode" and do not add mypy to `pyproject.toml`.

**Evidence:**

- `python -m mypy --version` returned `No module named mypy` in the current
  local environment.
- `pyproject.toml:42-48` does not include mypy in the dev extra.
- The phrase "strict mode" and "non-strict mode" describes two different
  contracts.

**Risk/impact:** The resulting `types.md` may be hard to reproduce or may be
based on an accidental global mypy version.

**Required action:** Specify an exact ad-hoc command and version capture, for
example:

```bash
python -m pip install "mypy==<version>"
python -m mypy --version
python -m mypy --strict src/groundtruth_kb/ > docs/reports/v0.4-baseline/types.raw.txt
```

Treat the non-zero mypy exit as data, not as a failed audit run. If the desired
baseline is non-strict, say so and remove `--strict`.

### 4. MEDIUM - Committing `coverage-html/` needs a firm decision before implementation

**Claim in proposal:** `coverage-html/` may be committed or gitignored; this is
left as a minor owner decision.

**Risk/impact:** A multi-file generated HTML tree can add avoidable repo noise
and make the review harder. It also conflicts with the stated preference for a
measurement report rather than generated artifacts.

**Required action:** Default to committing only Markdown and raw text/JSON
inputs needed for reproducibility. Commit `coverage-html/` only if the owner
explicitly chooses that before implementation.

### 5. LOW - Security/SBOM should be summarized, not expanded into a new report

**Evidence:** `.github/workflows/security.yml` runs Semgrep SAST and
`pip-audit`; it also writes `.quality/sbom.json` in CycloneDX JSON format before
uploading artifacts.

**Risk/impact:** A deep security audit would expand Phase 4A beyond the
approved quality-baseline scope.

**Required action:** Mention current Security workflow and SBOM generation in
`SUMMARY.md`, with latest workflow status if available. Do not add a full
security audit report unless a separate bridge proposal scopes it.
