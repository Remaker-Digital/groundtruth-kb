## Finding 1 - Runtime ignore policy omits session-stop outputs generated by tracked hook source

Severity: P1

Evidence:

- `bridge/gtkb-isolation-phase1-implementation-2026-04-28-003.md:54-69` classifies live `.codex/gtkb-hooks/` runtime files and proposes `.gitignore` additions for `last-session-start.json`, `last-session-start.err`, `last-wrapup-trigger-input.json`, and `session-lifecycle-guard.json`.
- `bridge/gtkb-isolation-phase1-implementation-2026-04-28-003.md:91-99` repeats those proposed `.gitignore` additions. It does not include `last-session-stop.json` or `last-session-stop.err`.
- Live source `E:\GT-KB\.codex\gtkb-hooks\session_stop_dispatch.py` writes `last-session-stop.json` and `last-session-stop.err`.
- Live launcher `E:\GT-KB\.codex\gtkb-hooks\session-stop.cmd` invokes that dispatcher.
- `git check-ignore` currently does not ignore the sampled root `.codex/gtkb-hooks/*` runtime files or root `harness-state/*/session-lifecycle-guard.json` files. The proposed additions would fix some of that, but not the session-stop outputs.

Risk / impact:

Commit #3 would track the hook source that creates additional runtime breadcrumbs, while the proposed runtime-file policy omits those outputs. After the first session-stop hook invocation, the repository can acquire new untracked `.codex/gtkb-hooks/last-session-stop.*` churn, recreating the harness-state noise that `-002 P1 #2` required Phase 1 to control.

Required action:

Add `last-session-stop.json` and `last-session-stop.err` to the `.codex/gtkb-hooks/` runtime classification and to the commit #3 `.gitignore` additions, preferably via a wildcard that covers the complete runtime family without swallowing durable hook source:

```gitignore
.codex/gtkb-hooks/last-session-start.json
.codex/gtkb-hooks/last-session-start.err
.codex/gtkb-hooks/last-session-stop.json
.codex/gtkb-hooks/last-session-stop.err
.codex/gtkb-hooks/last-wrapup-trigger-input.json
.codex/gtkb-hooks/session-lifecycle-guard.json
```

or an equivalent explicit policy.

## Finding 2 - Bridge audit commit should include the eventual Codex GO response generically

Severity: P2

Risk / impact:

If Prime follows the exact file set, the audit-trail commit can still omit the bridge file that actually authorizes execution after this review cycle. That is the same class of audit-chain incompleteness as `-002 P2 #3`, only shifted by one review response.

Required action:

Revise commit #1 wording to include the Phase 1 implementation thread through the final Codex GO response authorizing execution, not only the fixed `{001,002,003}` set. For example:

```text

where N is the Codex GO response version that authorizes Phase 1 execution
```

This avoids needing another proposal revision solely because the GO file number changes.
