## Summary

Add the generated hot-path projection of the MemBase `harnesses` registry
table, together with its generator and a DB-independent reader. The projection
is a flat JSON file (`harness-state/harness-registry.json`) regenerated from the
authoritative `harnesses` table; the generator lives in the `groundtruth_kb`
package (DB side); the reader lives in `scripts/` (DB-free, hot-path side).

This is work item WI-3338 (A2), the second work item of the
`PROJECT-HARNESS-REGISTRY-REFACTOR` sub-project. It implements
`REQ-HARNESS-REGISTRY-001` FR5: a generated flat projection serves the
SessionStart hot path so harness identity and role resolution never depend on
DB availability, schema currency, or lock contention. WI-3337 (the `harnesses`
table) is VERIFIED and supplies the authoritative store this projection is
derived from.

The change is strictly additive — two new modules, one new generated data file,
and two new test files. It does NOT touch `harness-state/role-assignments.json`
or `harness-state/harness-identities.json`, does NOT migrate any existing
reader, and does NOT change any existing behavior.

## Scope

In scope:

1. A new generator module `groundtruth-kb/src/groundtruth_kb/harness_projection.py`
   that reads the `current_harnesses` view (via `KnowledgeDB.list_harnesses()`),
   builds the projection document, and atomically writes
   `harness-state/harness-registry.json`. Exposes a pure `build_projection()`
   function, a `generate_harness_projection()` writer, a path resolver honoring
   a `GTKB_HARNESS_REGISTRY_PATH` environment override, and a `__main__` entry
   so the projection can be regenerated by command.
2. A new DB-independent reader module `scripts/harness_projection_reader.py`
   exposing `load_harness_projection()`. It imports only the Python standard
   library — no `groundtruth_kb` import, no DB connection — so it is safe to
   call at SessionStart before any DB connection exists. It mirrors the
   defensive load pattern already used by `scripts/harness_roles.py`
   (`load_role_assignments`): a missing or malformed file yields a normalized
   empty document rather than an exception.
3. The generated projection file `harness-state/harness-registry.json`,
   produced by running the generator against the current `harnesses` table and
   committed so the projection is fresh-clone-safe. Because the `harnesses`
   table is currently empty (population is WI-3342), the initial projection
   carries an empty `harnesses` list with valid schema metadata.
4. Generator unit tests in `groundtruth-kb/tests/test_harness_projection.py`.
5. Reader unit tests in `platform_tests/scripts/test_harness_projection_reader.py`,
   including a DB-independence assertion.

Out of scope:

- Seeding the `harnesses` table from the current JSON state — that is WI-3342
  (migration). The projection generated here reflects the empty table.
- Migrating the 8+ existing readers of `harness-state/role-assignments.json`
  and `harness-state/harness-identities.json` to consume the projection — that
  is WI-3342. No existing reader is modified here.
- Wiring automatic projection regeneration into the `gt harness` CLI mutating
  verbs — that is WI-3340. The generator is delivered here as an importable
  function plus a `__main__` entry; WI-3340 calls it after each mutation.
- The four-state lifecycle FSM (WI-3339), the `gt harness` CLI group (WI-3340),
  role portability (WI-3341), the ADR new version (WI-3343), and data-driven
  dispatch (WI-3344).
- Retiring `harness-state/role-assignments.json` and
  `harness-state/harness-identities.json` — per the phased migration that is
  the last step, after readers are cut over.

## Implementation Plan

1. Add `groundtruth-kb/src/groundtruth_kb/harness_projection.py`:
   - `PROJECTION_SCHEMA_VERSION = 1` and
     `HARNESS_REGISTRY_RELATIVE_PATH = Path("harness-state") / "harness-registry.json"`.
   - `harness_registry_path(project_root, override=None)` — resolves the
     projection path, honoring an explicit override then the
     `GTKB_HARNESS_REGISTRY_PATH` environment variable, mirroring
     `scripts/harness_roles.py:role_assignments_path`.
   - `build_projection(harness_rows)` — a pure function. For each current-version
     harness row it projects `id`, `version`, `harness_name`, `harness_type`,
     `status`, `role` (decoded from the DB JSON-text column to a native list),
     `reviewer_precedence`, `invocation_surfaces` (decoded from JSON text to a
     native object or `null`), and `capabilities_ref`. It wraps the records in
     schema metadata: `schema_version`, `source_of_truth`, `description`,
     `generated_at` (UTC ISO-8601), and `harnesses` (the list of projected
     records ordered by `id`). No derived topology field is stored (FR4).
   - `generate_harness_projection(db, project_root, *, projection_path=None)` —
     calls `db.list_harnesses()`, `build_projection(...)`, and atomically writes
     the JSON document with the temp-file + `os.replace` pattern and
     `indent=2, sort_keys=True`, matching `write_role_assignments`. Returns the
     written path.
   - `main()` plus an `if __name__ == "__main__"` guard — opens the default
     `KnowledgeDB`, regenerates the projection for the current project root, and
     prints the written path. This is the by-command regeneration entry; WI-3340
     will additionally call `generate_harness_projection` after each mutating
     `gt harness` verb.
2. Add `scripts/harness_projection_reader.py`:
   - `HARNESS_REGISTRY_RELATIVE_PATH` and `harness_registry_path()` resolver
     (env override `GTKB_HARNESS_REGISTRY_PATH`), consistent with the generator.
   - `load_harness_projection(project_root, *, projection_path=None)` — reads
     and JSON-parses the projection; on `FileNotFoundError`, `JSONDecodeError`,
     or `OSError` it returns a normalized empty document
     (`schema_version`, `source_of_truth`, `harnesses: []`), mirroring
     `load_role_assignments`. The module imports only the Python standard
     library — it does NOT import `groundtruth_kb` and never opens the DB.
3. Generate `harness-state/harness-registry.json` by running the generator
   against the current (empty) `harnesses` table, and commit it. The committed
   file carries valid schema metadata and an empty `harnesses` list, reflecting
   the empty table; WI-3342 regenerates it after the table is seeded.
4. Add `groundtruth-kb/tests/test_harness_projection.py` (generator tests — see
   Spec-to-Test Mapping).
5. Add `platform_tests/scripts/test_harness_projection_reader.py` (reader tests
   — see Spec-to-Test Mapping).

No existing table, view, method, module, hook, or caller is modified. The
generator's only DB use is the read-only `list_harnesses()` accessor delivered
by WI-3337.

## Spec-to-Test Mapping

`REQ-HARNESS-REGISTRY-001` FR5 (generated flat projection serving the
DB-independent SessionStart hot path) and the FR1 column-carry are verified by:

Generator tests (`groundtruth-kb/tests/test_harness_projection.py`):

- `test_build_projection_empty` — `build_projection([])` yields a document with
  valid schema metadata and an empty `harnesses` list (the initial-state case).
- `test_build_projection_carries_fr1_columns` — a harness row is projected with
  every FR1 field present; `role` and `invocation_surfaces` are decoded from
  JSON text to native list/object types.
- `test_generate_writes_projection_file` — `generate_harness_projection` writes
  the file at the resolved path; the on-disk JSON round-trips to the built
  document.
- `test_generate_reflects_current_versions` — with multiple versions of one
  harness id inserted, the projection carries only the latest-version record
  (it reads `current_harnesses`).
- `test_env_override_path` — `GTKB_HARNESS_REGISTRY_PATH` redirects the written
  path.

Reader tests (`platform_tests/scripts/test_harness_projection_reader.py`):

- `test_load_projection_roundtrip` — a projection written by the generator is
  loaded by `load_harness_projection`; the harness records compare equal.
- `test_loader_is_db_independent` — the reader module's source contains no
  `groundtruth_kb` import, and `load_harness_projection` succeeds against a
  fixture projection with no `groundtruth.db` present (FR5 DB-independence).
- `test_load_missing_file_returns_empty` — a missing projection file yields the
  normalized empty document, not an exception.
- `test_load_malformed_file_returns_empty` — a malformed projection file yields
  the normalized empty document, not an exception.

## Risks

- **Empty initial projection.** The committed projection carries an empty
  `harnesses` list because the table is empty pending WI-3342. *Mitigation:* no
  reader consumes the projection until WI-3342; an empty projection is inert,
  exactly as the empty `harnesses` table is inert. The empty-but-valid file
  still proves the generator and schema, and makes the artifact fresh-clone-safe
  from this work item onward.
- **Generated file drift.** `harness-state/harness-registry.json` is generated
  and git-tracked; a hand edit would diverge from the table. *Mitigation:* the
  file's `description` field states it is generated and must be regenerated, not
  hand-edited; WI-3340 wires regeneration into the CLI mutating verbs so the
  file stays in sync with the table.
- **JSON-column decode.** `role` and `invocation_surfaces` are stored as JSON
  text in the DB. *Mitigation:* `build_projection` decodes them with
  `json.loads`; tests assert the decoded native types; a `null`
  `invocation_surfaces` column projects as JSON `null`.

## Rollback

Remove the two new modules, the generated projection file, and the two new test
files. Nothing else references them, so removal is clean and leaves no dangling
caller. The `harnesses` table and its accessors are untouched by this work item.
