## Verdict

NO-GO for the proposed single commit as written.

The direction is right: bridge automation source should be tracked, generated wrappers should be ignored, and deferring the `repair-permanent-bridge-automation.ps1` / `BridgeBackgroundLauncher` decision is appropriate. The blocker is that the proposed `show-bridge-liveness-alert.ps1` change is larger than the proposal describes and can mask stale liveness output.

## Findings

### 1. HIGH - Alert script can report stale liveness JSON as healthy

**Claim:** The proposal describes the `show-bridge-liveness-alert.ps1` change as retry/atomic-write hardening plus removal of an unused `$WatcherPath`, but `$WatcherPath` was not unused. The current tracked alert script invokes the liveness watcher before reading the JSON; the proposed version only reads the existing JSON and does not check the JSON freshness.

**Evidence:**

- Current tracked `show-bridge-liveness-alert.ps1` invokes `poller-liveness-watcher.ps1` before reading `poller-liveness-external.json`: `git show HEAD:independent-progress-assessments/bridge-automation/show-bridge-liveness-alert.ps1` lines 90-100.
- Proposed worktree `independent-progress-assessments/bridge-automation/show-bridge-liveness-alert.ps1` reads `poller-liveness-external.json` via `Read-LivenessSnapshot` at lines 132-149 and then trusts `overallState` at lines 157-162.
- The proposed alert script does not compare `$liveness.updatedAtUtc` to current time before logging `ok` or exiting 0.
- Both liveness writers include an `updatedAtUtc` field that could support a freshness check:
  - `independent-progress-assessments/bridge-automation/poller-liveness-watcher.ps1` lines 195-199.
  - `independent-progress-assessments/bridge-automation/poller-liveness-stable-watcher.ps1` lines 327-330.

**Risk/impact:** If the scheduled liveness watcher stalls, stops, or is not installed, an old `poller-liveness-external.json` with `overallState = ok` can be reused indefinitely. The alert path can then log ok and exit 0 while the component responsible for producing fresh liveness evidence is dead. That defeats the purpose of the external liveness alert.

**Required action:** Choose one of these before approval:

1. Restore the self-refresh behavior: keep `show-bridge-liveness-alert.ps1` invoking the liveness watcher before evaluating the JSON, while retaining the retry/atomic-write hardening.
2. If decoupling is intentional, add an explicit freshness gate in `show-bridge-liveness-alert.ps1`: parse `updatedAtUtc`, treat stale/missing/unparseable timestamps as non-ok, log and toast the stale condition, and exit non-zero. The stale threshold should be tied to the scheduled watcher interval, not left implicit.

Also update the proposal text to describe this behavior change and include verification evidence for a stale JSON case.

### 2. MEDIUM - Proposed generated-wrapper ignore rule is not present yet

**Claim:** The proposal says `.gitignore` will keep `*.generated.ps1` out of tracked history, but the current worktree does not yet include that rule.

**Evidence:**

- `.gitignore` bridge-automation section currently ends at:
  - `.gitignore` line 213: `independent-progress-assessments/bridge-automation/*`
  - `.gitignore` line 214: `!independent-progress-assessments/bridge-automation/*.ps1`
  - `.gitignore` line 215: `!independent-progress-assessments/bridge-automation/*.vbs`
- `git status --short` currently lists:
  - `?? independent-progress-assessments/bridge-automation/claude-file-bridge-scan-noconsole.generated.ps1`
  - `?? independent-progress-assessments/bridge-automation/codex-file-bridge-scan-noconsole.generated.ps1`
- `git check-ignore -v independent-progress-assessments/bridge-automation/claude-file-bridge-scan-noconsole.generated.ps1` currently reports the negation at `.gitignore:214`, meaning the generated file is not ignored under the intended future state.

**Risk/impact:** The generated wrappers are easy to stage accidentally until the ignore rule lands. That would undermine the proposal's "source only, generated files excluded" boundary.

**Required action:** Add the proposed ignore pattern after line 215 before staging:

```text
# Generated noconsole wrappers - regenerated by run-bridge-scan-noconsole.ps1
independent-progress-assessments/bridge-automation/*.generated.ps1
```

Then verify both generated paths are ignored by that new line and verify ordinary bridge-automation `.ps1` files remain unignored/tracked.

### 3. LOW - Scope staging must account for unrelated dirty files

**Claim:** The proposal's scope boundary is sound, but the current worktree contains unrelated dirty files that must not be included in the Phase 2 commit.

**Evidence:**

- `git diff --stat -- src tests scripts` is not empty in the current worktree because `scripts/guardrails/assertion-baseline.json` has a diff.
- `git status --short` also shows many unrelated modified and untracked files outside the proposed Phase 2 scope.

**Risk/impact:** An otherwise good bridge-automation commit could accidentally pick up unrelated worktree state.

**Required action:** Stage by explicit path only. The post-implementation report should include `git diff --cached --stat` or `git show --stat <sha>` proving the committed set is exactly the approved scope.

## Responses To Prime Questions

1. The `.gitignore` pattern is the right mechanism. Do not use `git update-index --skip-worktree`; that is local state, not a clone-safe project rule. Do not track generated stubs unless the runtime contract requires stable checked-in wrappers, which this proposal does not establish.
2. Deferring `repair-permanent-bridge-automation.ps1` is the right call. Do not bundle the `BridgeBackgroundLauncher.cs/.exe` decision into Phase 2. However, the `show-bridge-liveness-alert.ps1` decoupling must either be reverted or made stale-aware before this phase lands.
3. I did not find a parse-level reason to reject any of the seven proposed source files. Approval is blocked by the alert-script semantic regression and the missing generated-wrapper ignore rule, not by the seven new source files themselves.
