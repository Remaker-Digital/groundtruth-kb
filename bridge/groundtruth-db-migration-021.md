### Phase 1i. SQLite sidecar disposition (new)

The live `tools/knowledge-db/` contains two SQLite WAL-mode sidecar files:
- `knowledge.db-shm` — 32,768 bytes (shared memory map, auto-generated by SQLite)
- `knowledge.db-wal` — 0 bytes (write-ahead log, empty = no pending transactions)

These are gitignored (`*.db-shm` at `.gitignore:107`, `*.db-wal` at `.gitignore:108`), runtime-generated, and contain no unique data.

**Safe deletion procedure:**

```bash
# Step 1: Ensure no active DB connections (close web UI, hooks, etc.)
# Step 2: Checkpoint the WAL to ensure no pending data (WAL is 0 bytes, but be explicit)
python -c "
import sqlite3
conn = sqlite3.connect('tools/knowledge-db/knowledge.db')
conn.execute('PRAGMA wal_checkpoint(TRUNCATE)')
conn.close()
print('WAL checkpoint complete')
"

# Step 3: Move the main database (Phase 1a — git mv)
git mv tools/knowledge-db/knowledge.db groundtruth.db

# Step 4: Delete orphaned sidecars (they reference the old DB path)
rm -f tools/knowledge-db/knowledge.db-shm tools/knowledge-db/knowledge.db-wal
```

**Owner-safety basis:** These files are:
1. Auto-generated by SQLite at runtime (not user-created data)
2. Already gitignored (never tracked or committed)
3. The WAL is 0 bytes (no pending transactions to lose)
4. Orphaned after the main DB moves — they cannot be used without `knowledge.db`
5. SQLite will regenerate new sidecars (`groundtruth.db-shm`, `groundtruth.db-wal`) at repo root on first access, and those are already covered by the existing `*.db-shm`/`*.db-wal` gitignore patterns

No owner approval needed for deleting auto-generated, gitignored, orphaned runtime artifacts with no pending data.
