## Summary

The project-VERIFIED-completion detection subsystem cannot recognize the `WI-AUTO-<SPEC-ID>` work-item ids that `groundtruth_kb.intake` (spec-intake confirm) auto-generates. Two byte-identical `_WORK_ITEM_LINE_RE` regexes scan bridge files for `Work Item:` metadata lines to determine which work items reached `VERIFIED`; the alternation accepts `WI-` followed by digits, plus `GTKB-*` and `WORKLIST-*`, but not `WI-AUTO-*`. Consequently a `WI-AUTO-*` work item whose bridge thread reaches `VERIFIED` is never counted as complete by project-completion logic: the per-authorization completion-readiness scan under-reports, and `ProjectLifecycleService.complete_project_authorization()` raises a false "not completion-ready" error. This proposal widens both regexes to also accept `WI-AUTO-[A-Z0-9-]+`. It does not change the spec-intake id scheme; it aligns the mechanization with the id forms the platform already produces.

This is the sibling concern that `bridge/gtkb-bridge-compliance-gate-wi-auto-regex-fix-001.md` (WI-3322) explicitly scoped out: that thread fixed the byte-identical defect in the bridge-compliance-gate hook and recorded the project-completion subsystem as a separate discovered defect for independent triage. WI-3335 is that triage.

## Defect Detail

The completion subsystem carries the same `_WORK_ITEM_LINE_RE` in two files, by a deliberate design decision - a cross-layer import of the `scripts/` scanner from package code was rejected as fragile, so the readiness check is mirrored rather than shared (documented in the `lifecycle.py` `_verified_work_items` docstring):

- `groundtruth-kb/src/groundtruth_kb/project/lifecycle.py` lines 20-23 - the regex; consumed at line 410 inside `ProjectLifecycleService._verified_work_items()`, which `complete_project_authorization()` (IP-3 of WI-3316) calls to gate authorization completion.
- `scripts/project_verified_completion_scanner.py` lines 33-36 - the regex; consumed at line 95 inside `verified_work_items()` (IP-1 of WI-3316), the read-only per-authorization completion-readiness scanner.

The alternation is `(WI-\d+|GTKB-[A-Z0-9-]+|WORKLIST-[A-Z0-9-]+)`. A `WI-AUTO-*` id fails every branch: `WI-\d+` requires a digit immediately after `WI-` (it gets the literal `A`), and the other two branches require a `GTKB-`/`WORKLIST-` prefix. So a `Work Item: WI-AUTO-SPEC-INTAKE-ABC123` line in a `VERIFIED` thread is not matched, the id is never added to the verified set, and:

- `verified_work_items()` under-reports - `completion_ready()` marks an authorization not-ready even when its only outstanding work item is a `WI-AUTO-*` id with a `VERIFIED` thread.
- `complete_project_authorization()` raises `ProjectLifecycleError("... not completion-ready ...")` for the same authorization, blocking a legitimate completion.

The `WI-AUTO-*` id form is generated by `groundtruth_kb.intake._stable_work_item_id_for_spec()`: `slug = re.sub(r"[^A-Za-z0-9]+", "-", spec_id.upper()).strip("-")` then `f"WI-AUTO-{slug}"`. The `.upper()` and the non-alphanumeric-run-to-hyphen substitution guarantee the slug is drawn strictly from uppercase letters, digits, and hyphen - exactly `[A-Z0-9-]+`. So `WI-AUTO-[A-Z0-9-]+` is an exact match for the generator's output set, not an approximation.

## Proposed Scope

### IP-1: Widen the Work Item alternation in the package module

In `groundtruth-kb/src/groundtruth_kb/project/lifecycle.py`, `_WORK_ITEM_LINE_RE` (lines 20-23): insert `WI-AUTO-[A-Z0-9-]+` as a new alternative ahead of `WI-\d+`. The alternation becomes `(WI-AUTO-[A-Z0-9-]+|WI-\d+|GTKB-[A-Z0-9-]+|WORKLIST-[A-Z0-9-]+)`. The `^Work Item:\s*` ... `\s*$` anchoring and the optional backtick wrappers are unchanged. The adjacent explanatory comment is updated to note the `WI-AUTO-` id form.

### IP-2: Widen the byte-identical regex in the completion-scanner script

Apply the identical edit to `scripts/project_verified_completion_scanner.py`, `_WORK_ITEM_LINE_RE` (lines 33-36), including the comment update. The two regexes remain byte-identical after the edit, preserving the deliberate mirror.

### IP-3: Regression test for the package module

In `groundtruth-kb/tests/test_project_artifacts.py` (the "IP-3 of WI-3316" section), add `test_complete_recognizes_wi_auto_verified_work_item`: seed a completion environment whose authorization covers a mixed work-item set `{WI-AUTO-SPEC-INTAKE-ABC123: VERIFIED, WI-8001: VERIFIED}` and assert `complete_project_authorization()` succeeds (the authorization transitions to `completed`). Against the un-widened regex this test fails - the `WI-AUTO-*` thread is not recognized, the authorization is judged not completion-ready, and the call raises. It passes only when the regex accepts `WI-AUTO-*`; the numeric `WI-8001` in the same set confirms the existing id forms still match. Derives from GOV-PROJECT-VERIFIED-COMPLETION-RETIREMENT-001.

## Risks and Rollback

- Risk: the widening accepts a malformed id. Mitigation: `[A-Z0-9-]+` matches only uppercase letters, digits, and hyphen - identical to the existing `GTKB-` and `WORKLIST-` branches; the `^Work Item:\s*` ... `\s*$` anchoring confines the match to a metadata line. The new branch and `WI-\d+` are mutually exclusive (a `WI-` id continues with either a digit or the literal `AUTO-`), so alternation order does not affect matching.
- Risk: the two regexes drift apart. Mitigation: IP-1 and IP-2 apply the identical edit; verification step 3 confirms byte-identity.
- Rollback: revert the `WI-AUTO-[A-Z0-9-]+` alternative in both files and remove the IP-3 and IP-4 tests. The change is a pure additive alternation; rollback restores prior behavior exactly.
