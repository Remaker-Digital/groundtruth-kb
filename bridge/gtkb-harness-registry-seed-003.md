## Implementation Notes

- `scripts/seed_harness_registry.py` reads `harness-state/harness-identities.json`
  (harness name to durable id) and `harness-state/role-assignments.json` (id to
  role-set and harness type), joins them on the durable id, and for each
  harness not already in the table inserts a row via `KnowledgeDB.insert_harness`
  at `status = active`. It then regenerates the FR5 projection via the
  WI-3338 `harness_projection.generate_harness_projection` generator.
- The migration is idempotent — a harness id already present in the table is
  skipped — verified by `test_seed_is_idempotent`.
- Per the proposal's interpretation note 1, the harnesses are seeded at
  `status = active` (migration semantics: importing already-live harnesses);
  the FR2 FSM governs future transitions from the seeded state.
- Per interpretation note 2, this slice only populates the table;
  `harness-state/role-assignments.json` and `harness-state/harness-identities.json`
  remain authoritative for role/identity resolution until the WI-3342 Slice B
  reader migration.
- Per the `-002` GO implementation note, the script bootstraps
  `groundtruth-kb/src` onto `sys.path` so the `groundtruth_kb` package imports
  resolve regardless of which interpreter runs the migration.
