### Finding 2 (HIGH) — `interrogate --output-format json` doesn't exist

**Codex evidence:**
- `python -m interrogate --help` lists `--output FILE` but NO `--output-format`
- Running the proposed command fails with `Error: No such option: --output-format (Possible options: --badge-format, --output)`
- `interrogate` not currently installed in the local Python environment until Codex installed it in a temp target for validation

**Acknowledged.** I assumed `--output-format json` from memory; interrogate has no such flag.

**Revised docstring audit approach — use the interrogate Python API directly:**

```python
from pathlib import Path
from interrogate import config, coverage

interrogate_conf = config.InterrogateConfig(
    ignore_init_module=True,
    ignore_init_method=False,
    ignore_magic=True,
    ignore_module=False,
    ignore_private=False,
    ignore_property_decorators=False,
    ignore_nested_functions=False,
    ignore_nested_classes=False,
    ignore_setters=False,
    ignore_regex=[],
    include_regex=[],
    fail_under=0,    # no threshold — audit only
    verbose=2,
    quiet=False,
    omit_covered_files=False,
    whitelist_regex=[],
)
cov = coverage.InterrogateCoverage(
    paths=[str(Path("src/groundtruth_kb"))],
    conf=interrogate_conf,
)
results = cov.get_coverage()
# results is an InterrogateResults object with .files[], .total, .covered, .missing, etc.
```

I then write a small audit script (`scripts/audit_docstrings.py`) that:
1. Runs the above
2. Categorizes results into public API (based on `__all__` export or method guide mention) vs private
3. Writes `docs/reports/v0.4-baseline/docstrings.md` as formatted Markdown
4. Records interrogate version via `interrogate.__version__` in the report

The audit script is committed as part of Phase 4A so the baseline can be re-generated by future sessions.

**Record the interrogate version used:** the script prints and records `interrogate.__version__` in both the report header and the commit message.

**Install requirement:** interrogate is already in `[dev]` extras per `pyproject.toml:42-47`. No new dependency needed.

### Finding 3 (MEDIUM) — mypy plan underspecified

**Codex evidence:**
- `python -m mypy --version` → `No module named mypy` (not installed locally)
- `pyproject.toml:42-48` dev extra does not include mypy
- The phrase "strict mode" and "non-strict mode" describes two different contracts

**Acknowledged.** The proposal was ambiguous about install, version, and invocation.

**Revised mypy audit plan — explicit and reproducible:**

```bash
# Step 1: Ensure mypy installed at a pinned version (ad-hoc, NOT committed to pyproject.toml)
python -m pip install --quiet "mypy==1.13.0"
python -m mypy --version
# Expected: mypy 1.13.0 (compiled: yes)

# Step 2: Run in strict mode, capture output including non-zero exit
set +e
python -m mypy --strict src/groundtruth_kb/ > docs/reports/v0.4-baseline/types.raw.txt 2>&1
MYPY_EXIT=$?
set -e
echo "mypy exit code: $MYPY_EXIT"

# Step 3: Write types.md summary from types.raw.txt
python scripts/audit_types.py docs/reports/v0.4-baseline/types.raw.txt docs/reports/v0.4-baseline/types.md
```

Commit includes:
- `docs/reports/v0.4-baseline/types.raw.txt` (the unprocessed mypy output, for reproducibility)
- `docs/reports/v0.4-baseline/types.md` (the categorized markdown summary)
- `scripts/audit_types.py` (the small analysis script)

**Version pinning:** mypy 1.13.0 is the current stable release as of 2026-04-14. The pin ensures Phase 4B (future bridge round) can replay the exact measurement. If mypy 1.14+ changes error counts, Phase 4B can compare the delta.

**Strict vs non-strict decision:** **Run with `--strict`** for the audit. The goal is to enumerate every error a fully-strict mypy would raise. Phase 4B can then triage the list and decide which to fix. Running non-strict first would undercount and bias the report toward "not that bad".

**The `types.raw.txt` file is expected to contain error lines.** Non-zero exit is data, not failure. The script captures exit code in the report.
