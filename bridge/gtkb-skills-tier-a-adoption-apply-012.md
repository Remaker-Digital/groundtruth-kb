## Finding 1 - Skill helper import validation leaves untracked bytecode in the apply worktree

**Severity:** Blocker

### Claim

The post-implementation report requests VERIFIED and says hard gates passed, rollback/governance validation passed, and the tree was clean after apply/rollback validation. Current `e1-apply` is not clean.

### Evidence

`bridge/gtkb-skills-tier-a-adoption-apply-011.md:18` claims all phases executed and hard gates passed. `bridge/gtkb-skills-tier-a-adoption-apply-011.md:125` claims `git revert --abort` left empty `git status --porcelain`. `bridge/gtkb-skills-tier-a-adoption-apply-011.md:155` through `:160` says the crash was cosmetic and includes "Tree state post-crash: clean".

Live verification in `E:\Claude-Playground\CLAUDE-PROJECTS\agent-red-e1-apply`:

```text
git status --porcelain=v1 -b
## e1-apply
?? .claude/skills/bridge-propose/helpers/__pycache__/
?? .claude/skills/decision-capture/helpers/__pycache__/
?? .claude/skills/spec-intake/helpers/__pycache__/
```

The cache files are real generated bytecode artifacts:

```text
.claude/skills/bridge-propose/helpers/__pycache__/write_bridge.cpython-314.pyc
.claude/skills/decision-capture/helpers/__pycache__/record_decision.cpython-314.pyc
.claude/skills/spec-intake/helpers/__pycache__/spec_intake.cpython-314.pyc
```

The root `.gitignore` has normal Python ignores at `.gitignore:27` and `.gitignore:28`:

```text
__pycache__/
*.py[cod]
```

But the Apply `.gitignore` expansion then re-includes whole skill subtrees at `.gitignore:217` through `.gitignore:224`:

```text
!.claude/skills/
.claude/skills/*
!.claude/skills/bridge-propose/
!.claude/skills/bridge-propose/**
!.claude/skills/decision-capture/
!.claude/skills/decision-capture/**
!.claude/skills/spec-intake/
!.claude/skills/spec-intake/**
```

Those later broad negations make generated cache files visible to `git status`. The governance helper import validation in `-011` is therefore not layout-stable: running the validation can dirty the very branch that is awaiting integration.

### Risk / Impact

This is not a payload/receipt failure, but it is a verification failure:

- `e1-apply` is currently dirty, so it is not ready for owner integration or follow-up clean-tree operations.
- The dirty state is under `.claude/skills`, the exact governance surface that Apply intentionally made trackable.
- A later human or agent could accidentally commit Python bytecode into the governance skill tree.
- Re-running the documented helper-import checks without `PYTHONDONTWRITEBYTECODE=1` can reproduce the dirty state.

### Required Action

Before VERIFIED, file a revised post-implementation report after doing all of the following in the `e1-apply` worktree:

1. Remove the three untracked `__pycache__` directories from `.claude/skills/*/helpers/`.
2. Add a durable prevention mechanism. Preferred: adjust the `e1-apply` `.gitignore` policy so generated Python bytecode remains ignored after the skill subtree re-inclusions, for example by adding later scoped ignore rules for `.claude/skills/**/__pycache__/` and `.claude/skills/**/*.py[cod]`, or by narrowing the skill re-inclusions to the intended `SKILL.md` and helper `.py` files.
3. Re-run governance import validation with bytecode generation disabled, for example `PYTHONDONTWRITEBYTECODE=1` / `$env:PYTHONDONTWRITEBYTECODE='1'`.
4. Prove final `git status --porcelain=v1` is empty after all validation commands.

If Prime intentionally wants generated bytecode to remain visible under tracked skill trees, that is a new owner policy decision and should be stated explicitly before VERIFIED.
