## Implementation Plan

### New verb: `gt backlog add-work-item` (GOV-12/13 chain, fail-closed)

A new module `cli_backlog_add_work_item.py` composing existing governed primitives. Non-dry-run execution order is fail-closed so no orphan test can ever be persisted:

1. **Validate inputs** — all work-item fields (delegated to the existing `add_backlog_item` validation) PLUS the test fields (`--test-title`, `--test-type` ∈ {assertion,e2e,integration,unit,manual}, `--test-expected-outcome`, `--test-spec-id` defaulting to the work item's `--source-spec-id`) PLUS a REQUIRED `--test-plan-phase <phase-id>`. Missing `--test-plan-phase` (non-dry-run) → non-zero exit, no mutation.
2. **Validate the phase exists** — resolve `--test-plan-phase` to a current `test_plan_phases` row. Missing/invalid → non-zero exit, no mutation. (This pre-validation is what makes the later append unable to fail on a missing phase, eliminating the orphan-test window.)
3. **Resolve attribution** — fail-closed mutating resolver (same as `add_backlog_item`); unresolved → non-zero exit before any write.
4. **Create the work item** — reuse `add_backlog_item` (no duplicated allocation/attribution logic).
5. **Create the linked test (GOV-12)** — `db.insert_test(...)` with a monotonic `TEST-NNNN` id.
6. **Assign the test to the phase (GOV-13)** — read the current phase row, append the new test id to its `test_ids`, insert a new append-only phase version.

`--dry-run` performs steps 1-2 (including phase-existence validation) and reports the allocations without writing. Output: JSON with `work_item_id`, `test_id`, `phase_id`. The verb is registered under the existing `backlog` group in `cli.py`.
