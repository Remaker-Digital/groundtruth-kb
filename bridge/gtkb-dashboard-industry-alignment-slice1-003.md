## 1. Background

The Agent Red / GT-KB dashboard lives under `docs/gtkb-dashboard/`. An owner
S306 review identified three classes of gap: missing industry metrics (DORA,
SLO, flow, security), system-specific gaps (bridge swimlane, dual-subject
scoping, CI embed), and usability issues (landing page fragility,
accessibility, alert routing).

This bridge delivers the **narrowest subset** whose scope is fully supported
by current data and current generator ownership:

- **§A** — Progressive-enhancement landing page for `index.html`.
- **§C** — Per-panel `last_refreshed_at` via the dashboard generator.
- **§E** — Alert-routing skeleton under `provisioning/alerting/`.

Retention (`§B` in `-001`) is dropped because `MAX_HISTORY=200` already
bounds the authoritative history owner. DORA (`§D` in `-001`) is dropped
because the underlying telemetry does not yet support honest computation.

## 2. Slice 1 Deliverables (REVISED)

### A. Progressive-enhancement landing page (`docs/gtkb-dashboard/index.html`)

**Current gap:** Landing page is `<meta http-equiv="refresh" content="0;
url=http://127.0.0.1:3000/...">`. Visitors without local Grafana see a
frozen/broken redirect with no actionable content.

**Proposed change:** Replace with a static informational page that:

- Renders the **most-recent KPI snapshot** from `dashboard-data.json` on
  page load via `fetch` (graceful fallback text if the JSON is missing).
- Surfaces the **`generated_at`** timestamp with an age badge (green < 15
  min, yellow 15–60 min, red > 60 min).
- Links to the live Grafana URL with an explicit **"Open live dashboard"**
  button (not automatic redirect).
- Documents install/start commands (`install_local_grafana.ps1`,
  `start_local_dashboard.ps1`) inline for first-time visitors.
- Includes a `prefers-color-scheme: dark` CSS fallback.

**Scope:** single HTML file. No new JS dependencies. Baseline accessibility
checks: semantic headings, button-element anchors for live links, sufficient
color contrast.

### C. Per-panel freshness surfacing — via the generator

**Current gap:** Only the global "Refresh Age" stat shows data freshness;
individual stat panels have no per-query freshness signal.

**Proposed change (corrected routing):**

- Modify `scripts/gtkb_dashboard/generate_grafana_dashboard.py` so stat-panel
  builders include a `last_refreshed_at` secondary value sourced from the
  parent `dashboard_metadata` table (or the panel's own table when it has an
  `as_of` column).
- Run the generator; commit the regenerated
  `docs/gtkb-dashboard/grafana/dashboards/gtkb-dashboard.json` in the same
  commit so tests remain pinned against current output.
- Update `tests/scripts/test_gtkb_dashboard_grafana.py` panel-ordering and
  panel-content assertions to cover the new secondary value.

**Scope:** ~10 generator panel builders updated + generator run +
regenerated JSON + test update.

### E. Alert-routing skeleton (`provisioning/alerting/`)

**Current gap:** `provisioning/alerting/` exists but is empty. No alert
rules are defined; no notifier wiring.

**Proposed change:** Add three minimal Grafana alert-rule YAML files:
- `release-blockers.yaml` — fires when `release_blocker_count > 0`.
- `failing-ci.yaml` — fires when `failing_ci_count > 0`.
- `stale-data.yaml` — fires when `data_freshness_age_minutes > 60`.

Alerts route to the Grafana alert list only (no email/Slack/Teams notifier
— wiring is deferred to Slice 2 per the original scope). Include a
lightweight validator test (e.g., PyYAML load + minimal required-key check)
under `tests/scripts/test_gtkb_dashboard_alerting.py`.

**Scope:** 3 YAML files + 1 small test.
