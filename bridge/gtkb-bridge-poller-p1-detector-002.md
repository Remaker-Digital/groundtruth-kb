## Finding 1 - Parser comment handling is too narrow for the live INDEX

Severity: P1

### Evidence

The proposal says parser rules are:

- skip lines starting with `<!--` and blank lines;
- parse `Document:` lines;
- parse status lines;
- return errors for malformed lines.

Applying those rules to the live `bridge/INDEX.md` leaves 37 malformed lines. Examples:

```text
1: # Bridge Index
378:   Versions 001-009 and 011 of this thread were generated by a parallel
379:   OS Claude poller harness during S307 and were not persisted to this
...
390: -->
454:   Versions 004-022 of this thread were generated by a parallel OS Claude
...
462: -->
518:   Slice 2 reconciliation S308: prior INDEX claimed VERIFIED at -006 with
...
532: -->
```

The multi-line HTML comment bodies in `bridge/INDEX.md` do not start with `<!--`; only the opening line does. A line-by-line skip rule that only ignores lines starting with `<!--` will parse the comment body and closing `-->` as malformed data.

### Risk / Impact

The detector would degrade immediately on the authoritative live queue, or the proposed `test_parser_against_live_index_md()` would fail. This defeats the main purpose of P1: reliable low-cost parsing of the canonical bridge state.

### Recommended Action

Update the parser contract to:

- skip Markdown heading/comment preamble lines such as `# Bridge Index`;
- support full HTML comment blocks from `<!--` through the matching `-->`, including multi-line bodies;
- add a fixture containing the current INDEX comment-block patterns.

## Finding 2 - File-existence validation conflicts with preserved historical INDEX entries

Severity: P1

### Evidence

The proposal's test list includes `test_parser_validates_referenced_file_exists`, and section 3.2 says status lines should validate the file reference.

The current live `bridge/INDEX.md` references 87 bridge files that are not present in this checkout. Examples from a live scan:

```text
bridge/gtkb-root-directory-migration-018.md
bridge/gtkb-root-directory-migration-017.md
bridge/gtkb-root-directory-migration-016.md
bridge/gtkb-root-directory-migration-015.md
bridge/gtkb-app-boundary-mechanism-audit-012.md
bridge/gtkb-membase-effective-use-umbrella-014.md
bridge/gtkb-dashboard-industry-alignment-slice2b-metrics-022.md
```

This is not hypothetical drift; `bridge/INDEX.md` also contains explicit historical notes about phantom or parallel-checkout bridge files retained for audit/reconciliation context.

### Risk / Impact

If missing referenced files are treated as parse errors, the detector cannot parse the live authoritative queue. If they are ignored silently, operators lose visibility into a known bridge-audit weakness.

### Recommended Action

Make file existence a non-fatal diagnostic, not a parser failure:

- `ParseResult.documents` should still include the document and current top status.
- `ParseResult.warnings` should include missing historical file references.
- Routing/diff should only require the current top file for an actionable transition to exist. Missing older historical files should not block detector operation.
- Add tests for both cases: missing old historical file = warning only; missing current top file = warning plus non-routable transition or parse diagnostic, depending on chosen policy.

## Finding 3 - Empty-checkpoint behavior would wake on every existing status without a safe baseline mode

Severity: P2

### Evidence

Section 3.3 says a missing or unparseable checkpoint is treated as empty, so every current top status counts as a transition on the next poll.

The current live index has 56 document entries. Treating an absent checkpoint as all-new would classify many old `GO`, `NO-GO`, `VERIFIED`, `NEW`, and `REVISED` statuses as fresh transitions on first startup.

### Risk / Impact

Even though P1 does not invoke harnesses, this checkpoint behavior becomes dangerous once later phases consume detector output. A first install, corrupt checkpoint, or reset could generate stale transition events for old work.

### Recommended Action

Add an explicit bootstrap mode:

- first run records a baseline checkpoint and audit event with `bootstrap=true`, but emits no routable transitions unless `--replay-existing` is explicitly requested;
- reset behavior should distinguish `reset checkpoint and baseline now` from `replay all current top statuses`.

## Required Revision

1. Extend the parser contract to handle the actual live INDEX preamble and multi-line HTML comment blocks.
2. Change bridge-file existence validation from hard parse failure to warning/diagnostic, with current-top-file policy clearly specified.
3. Replace empty-checkpoint "everything is a transition" default with a safe bootstrap mode.
4. Update the live-index regression test to assert parse success on current `bridge/INDEX.md` while preserving warnings for missing historical references.
