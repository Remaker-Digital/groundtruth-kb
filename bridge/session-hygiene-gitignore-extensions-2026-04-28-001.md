## 3. Item-by-Item Rationale

### 3.1 Items #5 + #6 — Dashboard auto-regen reports

**Files:**
- `docs/gtkb-dashboard/session-startup-report.md` (currently tracked; ~7.5 KB)
- `docs/gtkb-dashboard/session-wrapup-report.md` (currently tracked; ~3.4 KB)

**Why they exist:** Generated by `scripts/session_self_initialization.py` on every `SessionStart` and `Stop` lifecycle hook (per DELIB-0840 + DELIB-0841 / `GOV-SESSION-SELF-INITIALIZATION-001` / `GOV-SESSION-LIFECYCLE-PROACTIVE-ENGAGEMENT-001`).

**Why they shouldn't be tracked:** Same reason as the JSON dashboard data file (already gitignored at S317). They regenerate every session with timestamp-and-signal-count diffs that produce noise commits with no historical value. The current S319 session's diffs:

```
docs/gtkb-dashboard/session-startup-report.md:  3 lines changed (timestamp + 2 signal-count updates)
docs/gtkb-dashboard/session-wrapup-report.md:   1 line changed (timestamp only)
```

**Why now:** S319 session-start drift triage surfaced these as the only remaining auto-regen artifacts not gitignored after S317.

### 3.2 Items #7 + #8 — Harness-state hardlink aliases

**Files:**
- `.codex/agent-red-hooks/operating-role.md` (untracked; 319 bytes; hardlink-shared with `applications/Agent_Red/harness-state/codex/operating-role.md` — same inode)
- `.codex/agent-red-hooks/session-startup-preferences.json` (untracked; 46 bytes; hardlink-shared with `applications/Agent_Red/harness-state/codex/session-startup-preferences.json`)

**Why they exist:** S317 harness-state-authority-migration created the canonical home at `applications/Agent_Red/harness-state/codex/` (tracked) and retained the original `.codex/agent-red-hooks/` paths as hardlink aliases so existing Codex hook scripts (`session_start_dispatch.py`, `session_stop_dispatch.py`, etc., all at `.codex/agent-red-hooks/`) continue to work without modification. Editing either path edits both — they share inodes.

**Why they shouldn't be tracked:** Tracking both paths to the same inode would create dual-canonical confusion. The canonical home is already tracked; the aliases should be silently ignored. The S317 `.gitignore` addition correctly handled the runtime-breadcrumb subset (`last-session-start.json`, `last-session-start.err`, `last-wrapup-trigger-input.json`, `session-lifecycle-guard.json`) but missed `operating-role.md` and `session-startup-preferences.json`.

**Why now:** They've appeared as `??` (untracked) in `git status` since the S317 migration; S319 drift triage surfaced them as session-start hygiene noise.

### 3.3 Item #9 — MEMORY.md backup glob

**Pattern:** `memory/MEMORY.md.backup-*`

**Why it exists:** Per S309 GTKB-STARTUP-ENHANCEMENTS P1 (commit `3caa034d`), MEMORY.md was trimmed from 59,753 → 18,119 bytes (~10,400 tokens recovered). Before the trim, a backup was saved at `memory/MEMORY.md.backup-20260425-222126`. Future trim operations may create similarly-named backups.

**Why it shouldn't be tracked:** Single-use point-in-time backup. Has historical value as a one-time recovery artifact, but committing it would (a) not preserve history because git already has the pre-trim version in commit `3caa034d`'s parent, (b) create the same noise issue if future trims are performed, (c) bloat the repo with redundant content.

**Why now:** S319 drift triage. Filing as a glob pattern (`MEMORY.md.backup-*`) rather than specific timestamp so future backups are auto-ignored.

## 5. Codex Review Questions

1. **Pattern consistency with S317 telemetry-churn-policy.** Verify the `docs/gtkb-dashboard/session-{startup,wrapup}-report.md` regeneration mechanism is the same as `dashboard-data.json` (i.e., same hook, same lifecycle event trigger). If a different code path generates these files, confirm the gitignore policy applies.

2. **Hardlink-alias soundness.** The proposal asserts `.codex/agent-red-hooks/operating-role.md` and `applications/Agent_Red/harness-state/codex/operating-role.md` are hardlink-shared (same inode, link count 2). Verify with `stat` or equivalent. If they're separate files (not hardlinks) with synchronized content, the proposal needs a different gitignore strategy.

3. **Hook readability after gitignore.** Confirm that after gitignore + uncached, Codex's `session_start_dispatch.py` still reads the file correctly. The `.gitignore` only affects git tracking — file reads are unaffected. But worth confirming.

4. **Backup glob aggressiveness.** Pattern `memory/MEMORY.md.backup-*` matches any file matching that prefix. Confirm no legitimate backup naming convention would be unintentionally caught. (Existing pattern from S309 was timestamp-suffixed; future trims should follow the same convention.)

5. **Scope discipline.** This proposal bundles 5 items across 3 classes. If reviewers prefer 3 separate threads (auto-regen / hardlink-alias / backup-glob), the proposal can be split. Owner already chose bundled at session-start; flagging in case Codex disagrees.

6. **Phantom-INDEX risk.** Any concerns this thread will create new phantom-INDEX patterns when files are committed?
