## Findings

### Finding 1 - Docstring gate is based on an incorrect current measurement

Severity: Blocker

The proposal states current interrogate coverage is 51.4% and proposes a 51%
gate. At the inspected target HEAD with interrogate 1.7.0, the actual result is
65.3%. The Phase 4A baseline also says 60.42%, so 51.4% is not consistent with
either the baseline or current HEAD.

Risk/impact:

- `--fail-under 51` does not lock in the current Phase 4B.3 docstring state.
- A large docstring regression could pass while still staying above 51%.
- The stated "minimal headroom" rationale is false in the inspected checkout.

Required action:

- Re-measure `python -m interrogate src/groundtruth_kb/` on the intended CI
  platform before implementation.
- Set the threshold from that measured value with explicit headroom. If CI is
  close to the local 65.3%, a threshold around 64 is the right order of
  magnitude; 51 is not.
- Update the proposal's baseline table and rationale.

### Finding 2 - Coverage thresholds are not anchored to current line+branch values

Severity: High

The proposed coverage thresholds use stale Phase 4A values for at least
`config.py` and `cli.py`. Current line+branch coverage from the full suite is:
`db.py` 73%, `cli.py` 73%, `config.py` 94%, and `gates.py` 94%.
The proposed `config.py:80` threshold leaves 14 percentage points of slack, and
`cli.py:68` leaves 5 points. That is not "minimal headroom" for a lock-in gate.

Risk/impact:

- Phase 4B.6 would give the appearance of a ratchet while allowing meaningful
  regressions in public API coverage.
- Future maintainers would inherit thresholds whose rationale does not match
  the current codebase.

Required action:

- Rebaseline coverage on the same Python and OS used by `test-base`
  (`ubuntu-latest`, Python 3.11/3.12/3.13 if necessary).
- Pick per-file thresholds from current line+branch values, with stated
  headroom. If CI values are close to the local run, `config.py` should not be
  anywhere near 80.
- Update the coverage table and verification conditions accordingly.

### Finding 3 - The custom per-file coverage script is unnecessary scope

Severity: High

The proposal adds `scripts/check_per_file_coverage.py` plus a unit test, but
coverage.py can enforce a single-file threshold directly after pytest-cov has
created coverage data. Run one `coverage report` command per file, scoped with
`--include` and the file-specific `--fail-under` value.

Example pattern:

```text
python -m coverage report --include="*/src/groundtruth_kb/db.py" --fail-under=70
python -m coverage report --include="*/src/groundtruth_kb/cli.py" --fail-under=72
python -m coverage report --include="*/src/groundtruth_kb/config.py" --fail-under=92
python -m coverage report --include="*/src/groundtruth_kb/gates.py" --fail-under=92
```

Risk/impact:

- A new script creates more code to maintain for behavior coverage.py already
  provides.
- Reading `.coverage` SQLite directly couples project code to coverage.py's
  internal storage format.
- The script would add test and review surface to a CI-wiring round whose stated
  goal is minimal enforcement.

Required action:

- Revise Change 3 to use coverage.py's built-in `coverage report` commands.
- Omit `--cov-fail-under=0`; it is a no-op and the proposal already identifies
  it as noise.
- Only propose a custom script if a concrete built-in limitation is demonstrated.

### Finding 4 - Verification conditions and touchpoints need to match the revised design

Severity: Medium

The proposal's declared touchpoints include the custom script and its test, and
the post-implementation verification expects those files. If Change 3 is
revised to use built-in coverage commands, the touchpoints should shrink to the
two workflow files unless another justified file is introduced.

The verification condition also names "Security Scan", while the workflow name
in `.github/workflows/security.yml` is `Security`.

Required action:

- Revise the expected diff scope after removing the custom script.
- Use the actual workflow names in verification conditions: `CI`,
  `Docstring Coverage`, `Security`, and `SonarCloud`.
