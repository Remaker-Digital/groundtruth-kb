## Summary

Add a dashboard reachability probe to the fresh-session startup payload generated by `scripts/session_self_initialization.py`. Owner directive (S338, 2026-05-08): "session startup needs to verify that the dashboard is actually running and reachable." Live evidence at the time of filing: `http://localhost:3000/d/gtkb/groundtruth-kb-dashboard` returns `ERR_CONNECTION_REFUSED` (curl exit 7, ~2.25s timeout); `/api/health` likewise unreachable. The startup script currently has zero HTTP probes for the dashboard URL — the existing `live_probes` list at lines 5484-5511 covers GitHub Actions, GT-KB latest release, and GT-KB latest main only.

The proposed change adds a two-stage HTTP probe (Grafana `/api/health` + the dashboard URL itself) to the existing `live_probes` list. Failures degrade `validation_status` to `fresh_with_gaps` (warn-only) and surface new "Dashboard reachability" lines in the disclosure with the underlying error and a recovery hint. Startup continues regardless of probe outcome. Both Claude and Codex harnesses share `scripts/session_self_initialization.py`, so a single change covers both.

## Problem Statement

The current startup disclosure renders the Grafana dashboard URL as a hardcoded constant (`GRAFANA_DASHBOARD_URL = "http://localhost:3000/d/gtkb/groundtruth-kb-dashboard"`, line 136) without any liveness verification. When Grafana is not running — as in the S338 fresh session — the disclosure presents a working-looking link that resolves to `ERR_CONNECTION_REFUSED`. This violates the "implemented vs. intended" honesty discipline in `.claude/rules/operating-model.md` §3 ("claims about platform capabilities must distinguish implemented surfaces from intended surfaces") and weakens the live-evidence guarantee that `SPEC-PROJECT-DASHBOARD-KPI-LINK-001` requires of the startup dashboard surface.

The owner directive raises this from a known-but-tolerated gap to specification language under GOV-09 (Owner Input Classification Rule).

## Proposed Implementation

### Change set (single file)

**File:** `scripts/session_self_initialization.py`

1. Add a private helper `_probe_dashboard_reachability(source_label: str, url: str, timeout: float = 3.0) -> dict[str, Any]`:
   - Uses `urllib.request.urlopen(req, timeout=timeout)` (stdlib only — no new dependency).
   - Catches `urllib.error.URLError`, `socket.timeout`, `ConnectionError`, and generic `OSError`.
   - Returns the standard `live_probe` shape: `{"source": source_label, "kind": "live_probe", "required": False, "status": "queried" | "unavailable", "queried_at": <iso8601>, "detail": <http_status_or_url>, "error": <stringified_exception_or_None>}`.
   - Sets `User-Agent: gtkb-startup-reachability-probe/1.0` so Grafana logs identify the probe source.
2. Extend the `live_probes` list (current lines 5484-5511) with two new entries:
   - `source = "GT-KB dashboard /api/health probe"` (URL: `http://localhost:3000/api/health`).
   - `source = "GT-KB dashboard URL probe"` (URL: `GRAFANA_DASHBOARD_URL`).
3. Both probes feed the existing `live_probe_gaps` computation at line 5525, which automatically degrades `validation_status` to `fresh_with_gaps` when either probe is unavailable. No change to `validation_status` logic itself.
4. Add a new disclosure-rendering subsection under the existing "Live Project Dashboard" section that surfaces probe results:
   - "Dashboard reachability (Grafana `/api/health`): REACHABLE" or "UNREACHABLE — `<error>`".
   - "Dashboard reachability (dashboard URL): REACHABLE" or "UNREACHABLE — `<error>`".
   - Recovery hint (rendered only when at least one probe is unavailable): "Start Grafana with the documented platform command, then re-run startup or refresh the dashboard."
5. Expose `GRAFANA_HEALTH_URL` as a module-level constant alongside `GRAFANA_DASHBOARD_URL` so tests can patch it without monkey-patching `urllib`.

### What this proposal does **not** change

- The dashboard URL constant (`GRAFANA_DASHBOARD_URL`, line 136) — preserved as canonical per DELIB-1452.
- The `validation_status` enum or downstream consumers — the probe slots into the existing `fresh_with_gaps` path.
- The `dashboard_open_mode` / `_should_open_dashboard_on_session_start` logic — orthogonal.
- Any mutating governance state — this is a read-only HTTP probe.
- The release-gate or doctor checks (a follow-up bridge can wire the probe result into doctor; out of scope here per owner-confirmed warn-only semantics).

### Token / cost considerations (DCL-SESSION-STARTUP-TOKEN-BUDGET-001)

Two HTTP probes with 3s timeout each. When dashboard is up: both probes return in <100ms. When dashboard is down: ~6s of additional startup latency (one-time, fresh session only). Probe responses are not logged into token-cost context; only the per-probe `status` / `error` shape enters the rendered disclosure (~3 lines of additional text per startup, well within token-budget intent).

## Acceptance Criteria

1. Both new probes appear in the `live_probes` list of every fresh-session payload, regardless of dashboard reachability.
2. When either probe is unavailable, `validation_status == "fresh_with_gaps"` and the offending probe sources appear in `live_probe_gaps`.
3. The rendered "Live Project Dashboard" section of the user-visible startup message contains a "Dashboard reachability" line for each probe and a recovery hint when at least one probe is unavailable.
4. Startup completes (full disclosure rendered, focus menu emitted) under all probe outcomes, including total dashboard outage.
5. No new Python dependency is introduced — stdlib only (`urllib.request`, `urllib.error`, `socket`).
6. All nine new tests pass; the existing `test_session_self_initialization.py` suite continues to pass.

## Risk and Rollback

### Risks

- **R1 (low):** A hung TCP connect could exceed the 3s timeout under specific network-stack stalls. Mitigated by `urllib.request.urlopen(..., timeout=...)` which enforces a hard timeout. Worst case: ~6s additional fresh-session latency; bounded.
- **R2 (low):** Probing `localhost:3000` from a container or remote development environment where Grafana lives elsewhere would always show UNREACHABLE. Acceptable — the disclosure remains accurate ("dashboard not reachable from this session"); follow-up work could parameterize the probe URL via env var if real-world remote setups appear.
- **R3 (very low):** Grafana logs an additional GET per session start. Mitigated by the `User-Agent` header making the probe distinguishable in logs.
- **R4 (very low):** Test mocks could drift from the real probe behavior. Mitigated by mocking at the `urllib.request.urlopen` boundary, which is stdlib and stable.

### Rollback

The change is contained to one file plus one new test file. Rollback = `git revert` of the implementation commit. No data migration, no schema change, no canonical-state mutation, no governance-state change. The probe is purely additive on the read path.
