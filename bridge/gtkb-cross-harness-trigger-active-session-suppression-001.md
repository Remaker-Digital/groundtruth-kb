## Claim

Add an active-session suppression gate to `scripts/cross_harness_bridge_trigger.py` so that the trigger does NOT spawn a fresh harness subprocess when the recipient harness already has an active session running. The gate uses a heartbeat lock file written by a new PostToolUse hook step on each harness side; the trigger reads the recipient's heartbeat and skips dispatch when the heartbeat is fresh (default freshness: 120 seconds).

This addresses an operationally-observed defect: when Codex files a NO-GO during an in-session Prime, the cross-harness trigger correctly detects the Prime-actionable signature change and dispatches a *fresh* Prime subprocess via `claude -p ...`. That subprocess is independent of the in-session Prime, runs with its own context, and produces parallel revisions that diverge from the in-session Prime's reasoning. Today's session observed this with `bridge/gtkb-bridge-poller-event-driven-replacement-slice-4-smart-poller-retirement-001-009.md` (REVISED-4 written by an auto-dispatched Prime, not by the in-session Prime).

The suppression makes Slice 2's signature-state dedup the *inner* loop (avoid redundant dispatches on unchanged signatures) and adds an *outer* "is anyone home" gate that prevents dispatch into an already-active counterpart.

## Why Now

The auto-dispatch parallel-revision problem is structural, not configuration:

- Slice 2 signature-dedup correctly prevents *redundant* dispatches when the per-recipient signature is unchanged.
- It does NOT prevent dispatches *during* an active session: when Codex legitimately files NO-GO, the Prime-actionable signature legitimately changes, and a fresh Prime subprocess legitimately spawns.
- After Slice 4 retires the smart-poller, the cross-harness trigger is the SOLE dispatch path. Every NO-GO/GO during an active session = a fresh subprocess. The noise compounds.
- This thread blocks Slice 4 + the SessionStart formalization thread per owner directive: "File suppression NEW now; pause Slice 4 + formalization NO-GO revisions until suppression GO+VERIFIED."

## Design

### Heartbeat hook script

`scripts/active_session_heartbeat.py`:

- Single-purpose: writes/touches `.gtkb-state/cross-harness-trigger/active-{role}-session.lock` with the current UTC timestamp.
- Invoked from PostToolUse hook in each harness on every Bash/Write/Edit (Claude) or Bash/apply_patch (Codex) tool use, plus Stop.
- Role determined by `--role {claude|codex}` argument supplied in the hook command.
- Fire-and-forget: always exits 0; failures logged to stderr but do not block tool use.
- Atomic write: tmp-file + rename to avoid partial reads.

### Trigger gate

`scripts/cross_harness_bridge_trigger.py` `run_trigger()` adds a new step BEFORE the dispatch path (and after the signature comparison):

```
For each recipient with a changed signature:
  Read .gtkb-state/cross-harness-trigger/active-{recipient}-session.lock mtime.
  If mtime is within freshness_seconds (default 120):
    Set last_result = "counterpart_active_session_present"
    Skip dispatch (no subprocess.Popen call)
  Else:
    Proceed with dispatch (existing behavior)
```

Critically, the gate fires AFTER the signature changes — so the `signature` field in dispatch-state.json is updated normally (preserves dedup correctness) but the SPAWN is skipped. When the active counterpart session exits and its lock goes stale, the next signature change (or a manual `Bridge` prompt that surfaces the actionable entries) will dispatch correctly.

### Freshness window

Default 120 seconds. Configurable via `GTKB_ACTIVE_SESSION_FRESHNESS_SECONDS` env var (read by the trigger; default 120).

### Reciprocal dispatch correctness

Walk-through of the round-trip Prime-NEW → Codex-GO → Prime-acts-on-GO with suppression:

1. Prime is in-session. Prime writes NEW. Prime's heartbeat hook fires → updates `active-claude-session.lock` (timestamp = now). Trigger fires → Codex-actionable signature changed → trigger reads `active-codex-session.lock` → stale (Codex is not running) → dispatches Codex.
2. Codex starts (auto-dispatched). Codex's heartbeat hook fires → updates `active-codex-session.lock` (timestamp = now). Trigger fires → Prime-actionable signature unchanged → no dispatch.
3. Codex writes GO. Codex's heartbeat hook fires → updates `active-codex-session.lock`. Trigger fires → Prime-actionable signature changed → trigger reads `active-claude-session.lock` → fresh (Prime is still in-session) → **SKIP dispatch** (the in-session Prime will pick up the GO when it next probes INDEX or owner sends a `Bridge` prompt).
4. Codex exits. Lock file goes stale within 120 seconds.
5. If owner now sends `Bridge` to in-session Prime, Prime probes INDEX, sees GO, processes the entry. No noise.
6. If a long pause occurs and the in-session Prime exits, lock goes stale. A subsequent event-driven dispatch (e.g., owner reopens Codex and Codex files something) would correctly dispatch Prime because Claude's heartbeat is now stale.

### What suppression does NOT prevent

- Manual owner-triggered `Bridge` scans always work; they read INDEX directly and don't go through the trigger script.
- Cross-harness invocation by other tooling (e.g., owner running `claude -p ...` themselves) is not gated; the gate only suppresses the trigger's `subprocess.Popen` path.

## Implementation Plan

### IP-1 --- Heartbeat hook script

Create `scripts/active_session_heartbeat.py`:

- argparse: `--role {claude|codex}` (required); `--state-dir <path>` (optional; default `<project_root>/.gtkb-state/cross-harness-trigger`).
- Write `<state-dir>/active-{role}-session.lock` with UTC timestamp (e.g., as a small JSON `{"updated_at": "..."}` or just an empty file with mtime).
- Fire-and-forget: catch all exceptions; log to stderr; exit 0.

### IP-2 --- Hook registration

Add heartbeat as a NEW hook step in each harness's PostToolUse + Stop matcher groups. Heartbeat must run BEFORE the trigger script in the same matcher group so the lock is fresh when the trigger reads it for the active session's own recipient (which is then irrelevant since the trigger checks COUNTERPART, not self).

`.claude/settings.json` deltas (heartbeat with `--role claude`):
- PostToolUse Bash: heartbeat + existing trigger
- PostToolUse Write|Edit: heartbeat + existing trigger
- Stop: heartbeat + existing trigger

`.codex/hooks.json` deltas (heartbeat with `--role codex`):
- PostToolUse Bash: heartbeat + existing trigger
- PostToolUse apply_patch: heartbeat + existing trigger
- Stop: heartbeat + existing trigger

### IP-3 --- Trigger gate

Modify `cross_harness_bridge_trigger.run_trigger()`:

1. Read `freshness_seconds` from env var `GTKB_ACTIVE_SESSION_FRESHNESS_SECONDS` (default 120).
2. After computing per-recipient signature, BEFORE calling `_spawn_harness`, check counterpart's heartbeat:
   - For recipient="prime", check `active-claude-session.lock` mtime.
   - For recipient="codex", check `active-codex-session.lock` mtime.
3. If mtime < freshness_seconds:
   - Set `recipient_state["last_result"] = "counterpart_active_session_present"`
   - Set `recipient_state["last_skipped_reason"] = f"counterpart heartbeat fresh ({age:.0f}s ago)"`
   - Skip the spawn; results dict records `{"launched": False, "reason": "counterpart_active_session_present"}`.
4. If mtime is stale or lock is absent: proceed with existing dispatch path.

The signature field in dispatch-state.json is updated EITHER WAY (signature dedup correctness preserved); only the spawn is gated.

### IP-5 --- Stop hook awareness

The Stop hook on both sides currently invokes `cross_harness_bridge_trigger.py --stop-hook`. The heartbeat hook step runs first; the Stop trigger reads the (just-updated) heartbeat for self-side and the counterpart's heartbeat for dispatch decisions. Behavior is consistent with PostToolUse.

Stop hook output contract per Slice 3 GO: `{}` JSON to stdout for the trigger; the heartbeat script does not need to emit JSON because its hook step is not the Stop terminator.

## Risk / Rollback

**Risk surface:**

- **Risk: Stale heartbeat suppresses dispatch indefinitely.** A crashed harness leaves its lock file on disk; future dispatches to that role are suppressed until the file is manually deleted or overwritten. Mitigation: 120s freshness window means crashed harness self-clears within 2 minutes (the next dispatch attempt sees stale mtime and proceeds). Plus: the `gt project doctor` `_check_cross_harness_trigger` (per Slice 4 D4 design) can be extended to flag stale-lock-but-no-running-process as a WARN.
- **Risk: Heartbeat hook overhead.** Adds one file write per tool use. Mitigation: write is small (timestamp only); atomic-rename is fast; expected <5ms per fire.
- **Risk: Race between heartbeat write and trigger read.** Both fire on the same tool use. Mitigation: heartbeat hook runs BEFORE trigger in the same matcher group (sequential within the harness). The trigger is checking COUNTERPART heartbeat, not self, so the race is N/A for the gate decision.
- **Risk: Owner working interactively in Prime sees Codex's GO not auto-processed.** This is the intended behavior --- owner can manually `Bridge` to scan, or wait until exit. Mitigation: documented in §"What suppression does NOT prevent".

**Rollback:**

- Revert IP-1 (heartbeat script), IP-2 (hook registrations), IP-3 (trigger gate), IP-4 (tests).
- Lock files in `.gtkb-state/cross-harness-trigger/active-{role}-session.lock` go stale and unused; manual cleanup optional.
- Cross-harness trigger returns to Slice 3's behavior (signature dedup only, no active-session gate).
