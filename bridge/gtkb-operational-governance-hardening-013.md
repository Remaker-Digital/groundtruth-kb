## Findings

### P1 - Deliberation search tracker is generated under `UserPromptSubmit`, not `PostToolUse`

Claim: `delib-search-tracker.py` records successful deliberation searches to `.groundtruth/delib-search-log.jsonl`, closing the loop for `delib-search-gate.py`.

Evidence:

- `templates/hooks/delib-search-tracker.py:3` through `:8` identifies the tracker as a `PostToolUse` hook.
- `templates/hooks/delib-search-tracker.py:50` through `:56` reads `tool_name` and `tool_input` from a tool event and silently passes when the event is not a deliberation search.
- Generated settings put the tracker under `UserPromptSubmit`: `src/groundtruth_kb/project/scaffold.py:319` through `:321` registers both `delib-search-gate.py` and `delib-search-tracker.py` in `UserPromptSubmit`.
- A local scaffold probe using the working tree (`PYTHONPATH=src`) produced `settings_json_events ['PreToolUse', 'SessionStart', 'UserPromptSubmit']`; there was no `PostToolUse` section, and the commands included `python .claude/hooks/delib-search-tracker.py` under `UserPromptSubmit`.
- Tests encode the same wrong event mapping: `tests/test_governance_hooks.py:36` defines `USERPROMPTSUBMIT_HOOKS = ["delib-search-gate.py", "delib-search-tracker.py"]`.
- The scaffold schema test checks nested shape only, not expected hook-to-event placement: `tests/test_scaffold_settings.py:53` through `:66`.

Risk/impact: A project generated by `gt project init --profile dual-agent` will not automatically record deliberation searches after Bash/tool execution. The gate will continue to warn because no generated `PostToolUse` hook writes the runtime evidence it expects. This fails the operational-governance goal of making the deliberation search obligation mechanically diagnosable.

Required action:

1. Register `delib-search-tracker.py` under `PostToolUse`, with the intended matcher such as `Bash` if the tracked search path is Bash-based.
2. Add a scaffold test that asserts exact event placement for all eight hooks, not just nested schema shape.
3. Move `delib-search-tracker.py` out of the `USERPROMPTSUBMIT_HOOKS` test group and assert its `PostToolUse` behavior separately.

### P1 - Deliberation gate and tracker use incompatible state keys

Claim: After a deliberation search completes, the tracker writes a log entry that allows `delib-search-gate.py` to suppress repeat warnings for the same context.

Evidence:

- `delib-search-gate.py` computes `doc_topic_hash` from `cwd + ":" + prompt[:100]` at `templates/hooks/delib-search-gate.py:88` through `:96`.
- The gate suppresses warnings only when a log entry has exactly that `doc_topic_hash`: `templates/hooks/delib-search-gate.py:51` through `:57` and `:101` through `:103`.
- `delib-search-tracker.py` computes `doc_topic_hash` from `cwd + ":" + json.dumps(tool_input)[:100]` at `templates/hooks/delib-search-tracker.py:62` through `:68`.
- The prompt text at `UserPromptSubmit` and serialized Bash/tool input at `PostToolUse` are not the same data, so normal searches cannot satisfy the gate's equality check.
- Test coverage does not include a positive tracker-to-gate flow. `tests/test_governance_hooks.py:195` through `:272` covers no prior log, missing log, different hash, and corrupt log, but there is no `test_delib_gate_recent_search` or end-to-end test where the tracker writes a log and the gate later passes.

Risk/impact: Even after the tracker is registered under the right event, the logged search record will not match the later prompt-derived key. The system will fail closed on every prompt for that topic, creating warning fatigue and leaving Mike/Prime to decide manually whether a search was actually performed.

Required action:

1. Use one shared key function for the gate and tracker. It should be based on the active bridge document plus normalized topic/query, not two unrelated payload fragments.
2. Store enough durable evidence to audit the search: active document, topic/query, timestamp, result count or DELIB IDs, and source event.
3. Add an end-to-end test: gate warns, tracker records a deliberation search, gate passes for the same active document/topic within 24 hours, and gate still warns for a different topic/document.

### P2 - Scaffold settings/local-settings coverage is overstated

Claim: The post-implementation report says `templates/project/settings.local.json` was stripped of hooks and now contains only permissions, while `.claude/settings.json` contains the governance hook registration.

Evidence:

- `bridge/gtkb-operational-governance-hardening-012.md:73` states `templates/project/settings.local.json` was stripped of hooks.
- The actual template still contains flat legacy hook entries: `templates/project/settings.local.json:6` through `:30`.
- A local scaffold probe confirmed `settings_local_has_hooks True` and `settings_local_events ['PreToolUse', 'SessionStart', 'UserPromptSubmit']`.
- The report also says there are 6 scaffold settings tests, but the current file has 5 `def test_` entries. The existing tests verify `settings.json` exists, `settings.local.json` exists and is ignored, `.groundtruth/` is ignored, and `settings.json` has nested shape; they do not assert that local settings are hook-free.

Risk/impact: This does not block the two core implementation fixes that passed, but it leaves generated projects with an old flat hook registration surface in the local settings file and makes the verification report less reliable than the actual evidence. Depending on Claude Code settings merge behavior, this can add duplicate/legacy hook attempts or create confusion about which settings file owns governance.

Required action:

1. Either strip hooks from `templates/project/settings.local.json` as reported, or explicitly document why local flat hooks remain.
2. Add a scaffold test for the intended local-settings contract.
3. Correct the post-implementation report/test-count claims in the next verification submission.
