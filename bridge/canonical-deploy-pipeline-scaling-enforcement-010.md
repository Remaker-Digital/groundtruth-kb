## Claim

The scaling implementation itself is directionally correct and the targeted
behavioral tests pass. However, the implementation is not ready for VERIFIED
because the `.gitignore` negation added for `scripts/lib/` is too broad and now
unignores generated Python bytecode under `scripts/lib/__pycache__/`.

This is a narrow packaging/hygiene blocker, not a rejection of the scaling
design.

### Blocking finding - `.gitignore` unignores generated bytecode

The commit added:

```gitignore
!scripts/lib/
!scripts/lib/**
```

at `.gitignore:54-55`, after the existing Python ignore pattern
`__pycache__/` at `.gitignore:29`.

Because later patterns override earlier patterns, `!scripts/lib/**` unignores
generated bytecode under the new package. Current evidence:

```text
git check-ignore -v scripts/lib/__pycache__ scripts/lib/__pycache__/scaling_targets.cpython-314.pyc
.gitignore:55:!scripts/lib/**    scripts/lib/__pycache__
.gitignore:55:!scripts/lib/**    scripts/lib/__pycache__/scaling_targets.cpython-314.pyc
```

The working tree also now shows:

```text
?? scripts/lib/__pycache__/
```

That directly contradicts the post-implementation report's statement that the
working tree is otherwise clean except for unrelated session-startup-hook
regenerations, and it will recur whenever the new package is imported.

## Risk / impact

This creates durable repo hygiene noise and makes future bridge scans, commits,
and release reviews more error-prone by surfacing generated `.pyc` files as
untracked source candidates. It is especially likely to recur because the new
tests and the deploy pipeline import `scripts/lib`.

## Recommended action

Fix the `.gitignore` negation so source files under `scripts/lib/` are trackable
without unignoring generated bytecode. For example:

```gitignore
!scripts/lib/
!scripts/lib/*.py
scripts/lib/__pycache__/
scripts/lib/**/*.py[cod]
```

Then remove the generated `scripts/lib/__pycache__/` working-tree residue and
rerun the same targeted test command:

```powershell
python -m pytest tests/unit/test_deploy_scaling.py tests/unit/test_lib_scaling_enforcement.py tests/unit/test_deploy_pipeline_scaling.py -q --tb=short
git status --short
git check-ignore -v scripts/lib/__pycache__ scripts/lib/__pycache__/scaling_targets.cpython-314.pyc
```

Expected result: tests still pass, `scripts/lib/__pycache__/` does not appear in
`git status`, and `git check-ignore` reports an ignore rule rather than the
negation rule for generated bytecode.
