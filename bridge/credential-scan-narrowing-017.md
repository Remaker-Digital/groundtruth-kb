## Change Since v2 Report

### Finding 1 (P1): SUPERADMIN_PREVIEW_API_KEY hardcoded in deploy artifacts

**Problem:** `SUPERADMIN_PREVIEW_API_KEY=ar_stg_XAx07GhWPBVOanIjNK1DYhDQcXj3Q9RL`
remained hardcoded in 3 deploy files. This is an Agent Red credential (per
`PRODUCTION-ENV-CHANGES.md` which maps it to Key Vault), not a third-party
secret.

**Resolution:** Replaced with deploy-time placeholder in all 3 files:

- `scripts/deploy/production-gateway-generated.yaml:110-111`:
  `value: __SUPERADMIN_PREVIEW_API_KEY__`
- `scripts/deploy/_prod_env_vars.txt:35`:
  `SUPERADMIN_PREVIEW_API_KEY=__SUPERADMIN_PREVIEW_API_KEY__`
- `scripts/deploy/_prod_env_vars_clean.txt:35`:
  same placeholder

**`ar_stg_` key family decision:** `ar_stg_` is a legacy/one-off staging key
format. It does not appear anywhere in `src/`, `tests/`, or `scripts/` outside
the deploy artifacts. It is not generated by `secrets.token_urlsafe()` in the
current codebase. Rather than adding a new scanner family for a single
now-remediated value, `ar_stg_` is documented in the WI-DEPLOY-SECRETS
follow-up risk item alongside the remaining third-party credentials in those
same deploy files. If `ar_stg_` keys appear in future code, they should be
added to the scanner at that time.
