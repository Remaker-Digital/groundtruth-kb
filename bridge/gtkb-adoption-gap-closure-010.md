## Findings

### Finding 1 - P1: Default generated CI still fails because `gt assert` runs against intentionally failing seed-example assertions

**Claim in revision:** CI is generated by default for all profiles, with `minimal` and `standard` active steps including `gt --config groundtruth.toml assert` (`bridge/gtkb-adoption-gap-closure-009.md:163-169`, `:214-222`, `:314-324`).

**Evidence:**

- Current `gt project init` defaults `--seed-example` to `True` in `groundtruth-kb/src/groundtruth_kb/cli.py:581-582`, and `ScaffoldOptions.seed_example` also defaults `True` in `groundtruth-kb/src/groundtruth_kb/project/scaffold.py:47-48`.
- `scaffold_project()` seeds the DB with `include_example=options.seed_example` at `groundtruth-kb/src/groundtruth_kb/project/scaffold.py:107-108`.
- The example seed includes executable assertions against `src/tasks.py`: `create_task` at `groundtruth-kb/src/groundtruth_kb/seed.py:97-123` and `list_tasks` at `groundtruth-kb/src/groundtruth_kb/seed.py:126-143`.
- Current adopter docs explicitly say this assertion failure is expected: `groundtruth-kb/docs/start-here.md:137-154` shows `gt assert` returning `File not found: src/tasks.py` and explains that failures tell the user what to build next.
- Verification command on a fresh default `local-only` scaffold: `python -m groundtruth_kb --config groundtruth.toml assert` exited `1` with `FAILED: 2` and `File not found: src/tasks.py`.
- Control check: a fresh scaffold created with `--no-seed-example` returned `FAILED: 0` for `gt assert`, so the failure is specifically tied to default seed-example data.

**Risk/impact:**

The proposal says CI is generated by default and should be "out of the box," but the proposed active `gt assert` step will fail before the adopter has written any application code. That makes the generated default workflow a red build on first push for `local-only` and `dual-agent`, and likely for `dual-agent-webapp` as well. It also conflicts with current docs, which frame the seeded assertion failures as an interactive learning signal, not as a CI gate that must pass immediately.

**Required action:**

Revise G3 to define the seed-example/CI contract. Acceptable paths:

1. Keep seed examples default-on, but generated CI runs only governance assertions such as `gt assert --spec GOV-01` until the adopter enables full assertion CI.
2. Generate a minimal `src/tasks.py` implementation that satisfies the seeded task-tracker assertions whenever `seed_example=True`.
3. Change the default seed or CI behavior so a fresh default scaffold has green generated CI, and update docs accordingly.

Add acceptance tests that run the generated CI commands for each profile, not only YAML parsing and string checks.

### Finding 2 - P1: `full` tier still runs `pytest tests/` without requiring a generated passing test

**Claim in revision:** The `full` tier for `dual-agent-webapp` actively runs `pytest tests/ -v --tb=short`, and the proposal adds `src/__init__.py`, `pyproject.toml`, and `requirements.txt` stubs to make full-tier Docker CI non-broken (`bridge/gtkb-adoption-gap-closure-009.md:233-251`, `:300-310`, `:323-325`).

**Evidence:**

- The proposal's own generated-artifact baseline says current `dual-agent-webapp` has no `tests/` tree (`bridge/gtkb-adoption-gap-closure-009.md:202-210`).
- The proposed stub list for `dual-agent-webapp` includes only `src/__init__.py`, `pyproject.toml`, and `requirements.txt`; it does not include `tests/` or a smoke test (`bridge/gtkb-adoption-gap-closure-009.md:241-249`).
- Current `dual-agent-webapp` generation confirms the baseline: a fresh temp scaffold had `Dockerfile=True`, `pyproject.toml=False`, `requirements.txt=False`, `src=False`, and `tests=False`.
- Verification command from that fresh temp scaffold: `python -m pytest tests/ -q --tb=short` exited `1` with `ERROR: file or directory not found: tests/`.
- Existing current template evidence matches the risk: `groundtruth-kb/templates/ci/test.yml:40-41` runs `pytest tests/ -v --tb=short`, while `groundtruth-kb/src/groundtruth_kb/project/scaffold.py:285-319` copies webapp Docker/env/terraform templates but does not generate tests.

**Risk/impact:**

The `-009` plan fixes the Dockerfile copy prerequisites but leaves the active Python test step broken. A fresh `dual-agent-webapp` project would still get a failing full-tier workflow unless Prime silently adds tests beyond the approved scope or removes pytest during implementation. Either path changes the contract under review.

**Required action:**

Revise the full tier to choose one contract:

1. Generate a passing `tests/test_smoke.py` and any minimal app/module code it imports, then keep active `pytest tests/`.
2. Move pytest/mypy into the same advisory/commented block used by `standard` until the adopter adds `src/` and `tests/`.

Add an acceptance test that runs `python -m pytest tests/ -q --tb=short` in a generated `dual-agent-webapp` scaffold if pytest remains active.
